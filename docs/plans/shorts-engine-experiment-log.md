## 실험 연대기

### Exp 0 — 측정 체계 수립 (2026-07-21)
- **방법**: 채널이 실제 발행한 숏폼을 정답으로. 오디오 상호상관(`align.py`)으로 숏폼→롱폼 구간 역추적,
  `evaluate.py`가 Hit@N(IoU≥0.5)·경계오차로 채점. 성과는 절대 조회수 금지 — ±90일 동시기 채널 중앙값 대비 배수(tier).
- **결과**: "롱폼에서 쇼츠를 잘 뽑는가"가 측정 가능해짐. 매칭 정확도 양성 0.886~0.998 / 음성 0.403~0.601.

### Exp 1 — 엔진 길이 수정 전/후 (단일 홀드아웃, 2026-07-21)
- **방법**: 하하PD 경주 PC방(13분) 홀드아웃, 정답 3, IoU Hit@5. 홀드아웃은 학습 제외.
- **결과**: **Hit@5 0.00 → 0.67**. 원인은 "못 찾음"이 아니라 4~8초로 너무 짧게 잘라 IoU 붕괴 → 프롬프트를
  "셋업 포함 30~60초"로 재작성 + 30초 하한. 추천 길이 8초→45초.
- **발견 (중요)**: recommend가 temperature 0인데도 **실행마다 결과가 다름** — 같은 조건 5회가 Hit@5 0.00~0.67로 요동.
  → 단발 A/B로는 "학습 덕분"인지 "우연"인지 구분 불가.

### Exp 2 — 다중 홀드아웃 반복 A/B, IoU (2026-07-21)
- **방법 변경**: (1) 홀드아웃 3편(원정대5=18·원정대4=5·경주=3 = **26정답**)으로 확대,
  (2) 조건마다 **5회 반복 평균±σ**(변동 상쇄, `eval_repeat.py`), (3) 누수 없는 프로파일(홀드아웃 제외 22쌍 학습, conf 0.75).
- **지표**: Hit@5/10/20 (IoU≥0.5) micro-average. 판정 = Hit@10.
- **결과**:

  | 지표 | 학습 끔 | 학습 켬 | 상대개선 | 판정(차이 vs σ) |
  |---|---|---|---|---|
  | Hit@5 | 0.100 | 0.138 | +38% | 효과 있음 |
  | Hit@10 | 0.131 | 0.200 | +53% | **미확정** (σ 안) |
  | Hit@20 | 0.154 | 0.215 | +40% | 효과 있음 |

- **결론**: 학습은 픽을 개선하나(확정 +38~40%; Hit@10 +53%는 미확정) **소폭**. 절대 Hit@10=0.20 → **무인 발행 미달**.
  단일 홀드아웃의 낙관적 0.67은 표본 확대로 ~0.20이 실측임이 드러남.

### Exp 3 — 주제(내용) 기반 채점 + ④ few-shot + 로컬 (2026-07-22) 〔진행 중〕
- **방법 변경 (사용자 지적)**: IoU(같은 초)는 "엔진이 같은 순간을 숏폼감으로 잡았나"를 8초만 어긋나도 오답 처리 →
  **Topic-Hit@N** 도입(`eval_topic.py`): 발행 숏폼(제목+자막) vs 엔진 픽(제목+구간자막)을 **Gemini가 "같은 순간인가"로 판정**.
- **추가 변경**: (a) **④ few-shot** — 학습 규칙을 요약하지 말고 실제 고/저성과 구간 예시를 프롬프트에 원본으로 주입,
  (b) **로컬 실행** — 가정용 IP로 봇차단 회피, (c) **Gemini 호출 타임아웃 120s** — 한 호출이 행에 걸려 측정이 21분 스톨한 것 차단.
- **설계**: 3홀드아웃(26정답), **3조건 × 2회 반복 한 세션**(같은 변동 조건 head-to-head):
  ① off ② base(학습 규칙) ③ base+예시(④). 각 조건 recommend → topic_hits → Topic-Hit@5/10/20.
- **답하는 질문**: (i) off Topic vs 기존 IoU 0.20 — 기계적 메트릭이 얼마나 과소평가했나, (ii) off vs base — 학습 효과(공정),
  (iii) base vs base+예시 — ④가 추상 규칙보다 나은가.
- **결과 (2026-07-22 완료, 정직)**:

  | 조건 | Topic-Hit@5 | @10 | @20 | runs@10 |
  |---|---|---|---|---|
  | **off** (학습 없음) | 0.404±0.10 | **0.519**±0.06 | **0.692**±0.04 | [0.46, 0.58] |
  | base (학습 규칙) | 0.173±0.10 | 0.346±0.08 | 0.596±0.02 | [0.27, 0.42] |
  | base+예시 (④) | 0.231±0.04 | 0.346±0.00 | 0.558±0.06 | [0.35, 0.35] |

- **두 가지 큰 발견**:
  1. **IoU가 엔진을 크게 과소평가했었다** — off Topic@10=0.52 vs IoU에선 0.20 (**2.6배**). "같은 초" 요구가 엔진 실력을 깎았다. Topic으로 보면 엔진 top-10이 사람 편집자 선택의 **절반 이상**을 잡는다 — 훨씬 강한 성능.
  2. **학습이 오히려 성능을 떨어뜨린다 (Exp 2와 반대!)** — off > base > base+예시. Exp 2(IoU)는 학습 +38~53% 개선이라 했지만 Topic으로 재보니 **학습이 -33% 악화** (@10: 0.52→0.35). ④ few-shot도 도움 안 됨(base와 동일).
- **해석 (가설)**:
  - (a) 학습 프로파일(hookWeights 편중)이 **한 채널 편향**을 만들어 다양한 순간을 놓침. hookWeights가 잘 터진 훅에 편중해 사람이 고른 폭넓은 순간을 못 잡음.
  - (b) IoU와 Topic이 **다른 걸 잰다**: IoU는 "정확한 위치·길이" 보상, Topic은 "같은 순간이면 됨". 학습은 위치·길이엔 도움(Exp 2) BUT 주제 다양성엔 해.
- **결론**: **결과가 뒤집혔다.** Topic(제품 목적에 가까운 지표) 기준으로 학습 프로파일은 **끄는 게 나음**. Exp 2의 "학습 +38~53%"는 IoU 특성 때문일 가능성. ④ few-shot도 이 형태로는 효과 없음.
- **다음 (숫자가 정하는 다음)**:
  - Topic vs IoU 격차의 원인 규명 — 왜 학습이 IoU엔 도움 되고 Topic엔 해가 되나
  - 학습 형태 재고 — hookWeights를 다양성 유지 방식으로 재설계, 또는 학습을 후보생성 대신 **랭킹 소량 부스팅**으로만
  - 다른 채널로 재현성 확인 — 이 반전이 하하PD 특수인지 일반인지
- **정직한 한계**: (i) Topic 판정은 LLM 심사(약간 주관, IoU 같은 하드 객관지표 아님) → 두 지표 병기 필요, (ii) 홀드아웃 3편·정답 26·반복 2회 — 표본 여전히 작음. 반복 5회로 늘리면 σ 반으로 줄어 확정성 강화 가능(다음 작업).

## 발견 — 고컷 토크 콘텐츠는 장면 유닛을 코스화해야 (2026-07-22, 사용자 지적)

**관찰**: 실험이 특정 홀드아웃(경주 PC방, 13분 토크)에서 유독 느렸다. 원인 실측:

| 홀드아웃 | 장면수 | 영상 | 평균 장면 | 방식 |
|---|---|---|---|---|
| **경주** | **432** | 13분 | **1.8초** | 시각 마이크로컷 |
| 원정대5 | 197 | 61분 | 13.8초 | 자막 기반 |

**13분 토크 영상이 432개 마이크로컷** — 사람 A→B→C 카메라 전환마다 1.8초씩 끊고 각 프레임을 분석했다.
**이런 1.8초 컷의 프레임 분석은 무의미**(화면은 그냥 "다음 사람 말하는 중")하고, recommend가 432 유닛을
처리하느라 느리다. → **정답: 시각 컷이 아니라 대사(자막)로 세그먼트.** 경주를 자막 코스유닛으로 재구성하니
**432 → 38 (11배↓)**, 실험이 21분 스톨 → 3분 완료.

**설계 함의** (로드맵 §5-6 '풀 파이프라인 필요성'과 직결):
- 고컷/토크 콘텐츠 = 시각 장면감지·프레임 비전이 **낭비**. `fast` 모드(자막 세그먼트)가 정답.
- recommend 비용도 유닛 수에 비례 → 대사 기반 **코스 유닛(30~40초)**이 속도·의미 둘 다 이득.
- **다음**: 컷 밀도(장면수/분)가 높으면 자동으로 fast/코스 경로로 라우팅.

### Exp 4 — 컨셉 전환: 채널맞춤 → **일반화(잘 터진 쇼츠 요소)** + 크로스채널 검증 (2026-07-22) 〔진행 중〕
- **이전 결함 인지 (사용자 지적)**: Exp 2·3의 학습은 (a) **학습·검증 같은 채널**(하하)이라 오염, (b) 표본 26정답 × 2회로 σ가 커 결정 불가, (c) 감사 재실행에서 3조건 모두 같은 숫자(해상도 부족).
  → "채널맞춤"을 포기하고 "**일반적으로 잘 터지는 쇼츠 요소**"를 배우는 방향으로 재정의.
- **데이터 방식 변경**: 롱폼↔숏폼 매칭(71쌍, 오디오정렬 병목) → **하하 발행 숏폼 228편만** 학습 소스로.
  숏폼만 있으면 되므로 규모 확장 자유(다른 100만 채널도 학습 데이터 편입 가능).
- **학습 대상**: 숏폼 자체의 요소(자막·길이·훅) × 성과 tier(채널 log-조회수 3분위: high 76·mid 76·low 76 — 완벽 균형).
- **파이프라인**:
  - Phase 1a: 하하 숏폼 오디오 다운로드 (199/228 완료, 로컬 봇차단 0, 148MB).
  - Phase 1b: 각 숏폼 STT (Gemini 병렬, 진행 중).
  - Phase 1c: (오디오만 있으므로) 프레임은 후단계로 미룸 — 자막·길이·훅 학습부터.
  - Phase 2: high vs low 요소 대조 → 일반화 프로파일.
  - Phase 3: **다른 채널(ENA 등) 롱폼**에 하하학습 적용 → 그 채널 발행 숏폼 Topic-Hit 재현.
- **평가 지표 (사용자와 확정)**: 다른 채널의 발행 숏폼 재현율 (Topic-Hit@N). off vs 하하학습 on. Win-rate·간접이지만 즉시 가능.
- **정직한 함정 (미리 인지)**: (i) 다른 채널이 하하와 스타일 겹치면(예능·토크) 학습 덕분처럼 보일 수 있음 → 다양 채널 필요, (ii) 성과 tier=조회수라 자름 품질과 포장이 여전히 섞임, (iii) 하하가 100만 채널이라 학습 원칙이 소규모 채널에 통할지 별개.
- **Phase 1·2 결과 (2026-07-22 완료)**:
  - Phase 1a: 하하 숏폼 오디오 다운로드 **199/228** (fail 29 = 삭제·비공개), 148MB. 로컬 봇차단 0.
  - Phase 1b: STT **199/199 fail 0** (63분, 순차, tmp 충돌 회피).
  - Phase 2 학습 (표본 high 69·mid 69·low 69, dur≤180s 재필터):

    | 요소 | high | low | lift | 신호 강도 |
    |---|---|---|---|---|
    | **제목 어그로(n_hooks)** | 0.13 | 0.07 | **1.86** | 최강 |
    | 이모지 | 0.13 | 0.09 | 1.44 | 강 |
    | 인용부호 | 0.04 | 0.03 | 1.33 | 약 |
    | 제목 길이 | 28.6자 | 24.9자 | 1.15 | 약 |
    | 숏폼 길이 | 44초 | 54초 | 0.82 | 강(짧을수록) |
    | 자막 총량 | 357자 | 433자 | 0.82 | 강(적을수록) |

  - **일반화 원칙**: "짧고(45초), 대사 적고, 제목 자극적"인 하하 숏폼이 잘 터진다.
- **Phase 3 결과 (2026-07-22 완료, 크로스채널 A/B, 2회 실행)** ⭐:
  - ENA 정답지: 30 숏폼 × 5 롱폼 오디오매칭 → **7쌍 확보** (4 롱폼 커버, score 0.80~0.99)
  - Topic-Hit@10 (Gemini 내용 매칭, 각 조건 **2회 독립 실행**):

    | 조건 | 1회차 | 2회차 | 평균 | 범위 |
    |---|---|---|---|---|
    | off (프로파일 없음) | 0.571 | 0.857 | **0.714** | 0.571~0.857 (**요동**) |
    | **하하학습 on** | 0.857 | 0.857 | **0.857** | 0.857 (**안정**) |
    | Δ (평균) | | | **+0.143 (+20%)** | |

  - 롱폼별 (1회차): QNtoQ4zI8mc에서 off @10=0.0 → 하하학습 @10=1.0 (편집자 선택 재현). 2회차에선 off도 1.0 나옴.
- **결론 (Exp 4 최종, 두 발견 분리)**:
  - **① 평균 개선 확인 (약함)** ✅: 학습 켬 평균 @10=0.857 vs off 0.714 → +20%. 표본 7쌍·반복 2회라 통계적 확정은 아직.
  - **② 안정성 격차 (더 중요할 수 있음)** ⭐: **학습 켬은 두 번 다 0.857 (동일)**. off는 0.571~0.857 요동. 하하학습이 픽을 안정화하는 효과 — 이게 실전(예측가능성)에 더 큰 가치.
  - **크로스채널 실증**: Exp 3(같은 채널 학습·검증)에서 뒤집혔던 결론이, 채널초월 일반화 원칙 + ENA 검증으로 **"방향 확립"** 단계까지 진전. "학습이 다른 채널을 해치지는 않는다" + "일부 경우엔 확실히 낫다"까지는 실측.
- **정직한 한계**: (i) ENA 정답지 7쌍은 여전히 작음 — 매칭 확대(30 숏폼 × 더 많은 롱폼) 필요, (ii) 각 조건 1회만 실행 — 반복 σ 미측정, (iii) ENA도 예능이라 하하와 스타일 겹침 — 다른 도메인(드나드나·려진 등) 확장 필요, (iv) LLM 판정관 detail 저장했으니 감사 가능.
- **다음**: (a) ENA 매칭 확대(정답지 30쌍+), (b) 반복 3회+로 통계 확정, (c) 다른 도메인 채널 확장, (d) 시각 요소(오프닝 프레임·오버레이)도 학습에 통합.

## 실패·폐기·오판 실험 (정직성 기록)

> 이 섹션이 왜 있나: 실험은 실패하고 결론이 뒤집힌다. **뭐가 안 됐고 왜 안 됐는지**를 남겨야 같은 실수를 반복 안 하고, 보고서에도 "우리가 뭘 걸러냈나"를 쓸 수 있다.

### F1 — 종결어미 스냅 (P1 계획 → 폐기)
- **시도**: 한국어 문장 끝(습니다/네요/까요 등)에 클립 경계를 스냅. 계획서 P1 우선순위였음.
- **실측 결과**: 홀드아웃에서 Hit·경계오차 **모두 변화 없음** (11.4초 유지).
- **원인**: 남은 경계오차는 "문장 중간 컷"이 아니라 "엔진과 편집자의 클립 범위 차이"였음. ±3초 미세조정으로 11초 차이를 못 메움. 오히려 멀쩡한 경계를 애매하게 옮기기도.
- **결정**: 프로덕션 미배포·폐기. 대신 **발화(문장) 타임스탬프 스냅**(Exp 2 병행)이 실효 있어 그것만 배포.
- **교훈**: "좋아 보이는 아이디어"라도 숫자 안 나오면 버린다. 계획서 우선순위 ≠ 실효성.

### F2 — 단일 홀드아웃 Hit@5 = 0.67 (조기 환희 → 정정)
- **주장 (Exp 1)**: 엔진 길이 수정으로 Hit@5가 0.00 → **0.67**. "무인 발행 임박!"으로 오해할 뻔.
- **문제**: 홀드아웃 1편(경주 PC방)·정답 3건. 해상도가 0/0.33/0.67/1.0 4단계뿐.
- **재검증**: 같은 조건 5회 반복 → Hit@5 0.00·0.33·0.33·0.67·0.67 (평균 0.40, σ 0.30). **엔진 자체 변동에 완전히 묻힘.**
- **정정 (Exp 2)**: 홀드아웃 3편·정답 26건·5회 반복으로 확대 → 진짜 Hit@10 = **0.20**.
- **교훈**: 낙관적 초기 숫자는 반드시 표본 확대 + 반복 평균으로 재검증. **temp 0이라도 LLM 결과는 요동친다.**

### F3 — Exp 3 "학습이 성능을 -33% 악화" (뒤집힌 결론 → 신뢰불가 판정)
- **주장**: Topic 기반 A/B에서 off > base > base+예시 (@10: 0.52 → 0.35). "학습 프로파일 오히려 해로움."
- **감사 (사용자 지적 후)**: 경주 1편으로 재실행하니 3조건 모두 Topic-Hit@10 = **0.667 (동률)**. 정답 3건 = 해상도 4단계라 픽이 달라도 숫자 같음.
- **근본 결함 인지**:
  - (a) **학습·검증 같은 채널** (하하 22쌍 학습, 하하 3편 검증) → 채널 오염, 완전한 hold-out 아님.
  - (b) **표본 26정답 × 2회** → σ 0.06~0.10이 지표 차이(0.17)를 삼킴.
  - (c) LLM 판정관 detail 미저장 → 왜 매칭/미매칭인지 감사 불가.
- **결정**: "학습 악화" 결론 **철회 · 신뢰불가**. 방법론 결함으로 재분류.
- **교훈**: 뒤집힌 결론일수록 **먼저 실험 자체를 의심**. 채점 파이프라인 print를 stderr로만 보내면 나중에 감사 못 한다 → 결과 파일에 detail 병기.

### F4 — 클라우드 워커 A/B (21분 스톨)
- **시도**: 주제 기반 A/B를 클라우드 워커에서 실행.
- **문제**: Gemini 호출 하나가 응답 없이 무한 대기 → `call_with_retry`에 타임아웃 없어 파이프라인 전체 스톨. 21분간 진행 0.
- **원인**: `genai` 클라이언트가 timeout 파라미터를 안 받아 SDK 기본 무한 대기.
- **수정**: `_call_with_timeout` 데몬 스레드 + join(120s) — 초과 시 일시오류로 보고 재시도.
- **결정**: 이후 모든 실험은 **로컬 실행 + 결과 파일 저장 + 스톨 감지 워치독** 원칙.
- **교훈**: 외부 API 호출은 반드시 타임아웃 감쌈. SDK 기본을 믿지 말 것.

### F5 — 클라우드 다운로드 봇차단 (매칭 잡 실패 다발)
- **시도**: 클라우드 워커에서 유튜브 대량 다운로드(매칭용). 데이터센터 IP.
- **결과**: 30편쯤 받으니 YouTube "Sign in to confirm you're not a bot." 봇차단. 매칭 잡 다수 실패.
- **원인**: 데이터센터 IP는 봇 취급. 쿠키(2.5KB짜리 약한 것)도 못 뚫음.
- **수정 시도들**:
  - 쿠키 강화 → 여전히 봇차단 재발 (근본은 IP)
  - 다운로드 스로틀 → 속도만 느려짐
  - **로컬 전환 (가정용 IP)** → 봇차단 0. 하하 숏폼 199편 · ENA 30편 전부 성공.
- **결정**: **연구용 다운로드는 로컬** (실서비스는 업로드 기반이라 무관 — [infra.md](../ops/infra.md) 참고).
- **교훈**: 다운로드 병목의 근본은 IP·인증이지 코드가 아니다.

### F6 — 병렬 STT tmp 파일 충돌 (초기 STT 결과 오염)
- **시도**: STT 6개 병렬 실행 (하하 숏폼 199편).
- **결과**: 12편 완료 시점에 로그가 "STT window @134,000s failed" 만 반복 (137,000초?). 실제 오디오는 24-30초짜리.
- **원인**: `core.asr._to_wav`가 tmp 파일명에 `os.getpid()`만 씀 → 병렬 스레드가 **같은 파일을 서로 덮어써서** 이상한 duration 감지 (여러 파일이 이어붙어진 값).
- **수정**: **순차 실행 강제** (workers=1). 각 편 짧으니 순차라도 15-20초/편으로 감당 가능. 이전 12편 삭제 후 재실행.
- **교훈**: 병렬 처리 시 tmp 파일명은 반드시 스레드 고유(pid + 스레드 id + 랜덤). 병렬 이득 있는지 먼저 확인.

### F7 — YouTube API `isShort` 플래그 신뢰 문제 (학습 데이터 오염 직전)
- **시도**: 하하 숏폼 228편을 `isShort or dur<=180` 조건으로 필터해 학습 소스로.
- **문제 감지**: Phase 2 학습 실행하니 **low tier 평균 길이 527초** (8.7분?!). "숏폼"에 롱폼 다수 섞임.
- **원인**: YouTube Data API의 `isShort` 플래그가 부정확 (라이브 스트림·재편집·API 판단 오류).
- **수정**: **실측 dur ≤ 180초**만 재필터. 228 → 207편 (21편 이상치 제거). tier 균형도 76/76/76 → 69/69/69로 정정.
- **교훈**: 외부 API 플래그를 그대로 믿지 말 것. 학습 데이터는 실측치로 재검증 필수. **초기 통계에서 이상한 숫자가 나오면 그게 신호다** (n_hooks lift 1.86보다 dur_sec 이상치가 먼저 알려줌).
