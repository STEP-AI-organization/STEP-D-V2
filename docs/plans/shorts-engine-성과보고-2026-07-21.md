## 2. 성과 — 이룬 것

### 2-1. 측정 체계 — "잘 뽑는가"를 숫자로 ✅

채널이 **실제 발행한 숏폼**을 정답지로 삼았다. 사람이 고른 것이므로 가장 신뢰할 수 있는 정답이다.

| 만든 것 | 역할 |
|---|---|
| 오디오 매칭 (`align.py`) | 발행 숏폼이 롱폼 **어느 구간**에서 잘렸는지 소리로 역추적 (Gemini 없이 CPU) |
| 채점 (`evaluate.py`) | 추천 구간 vs 정답 구간을 **Hit@N·경계오차**로 채점 (IoU≥0.5) |
| 성과 정규화 | **조회수 절대값 금지** — ±90일 동시기 채널 중앙값 대비 배수. 3년 전 50만과 지난달 20만을 같은 잣대로 비교하는 오류를 구조적으로 차단 |

매칭 정확도 실측: 진짜 매칭 0.886~0.998 vs 무관 0.403~0.601 → 간격이 넓어 오탐·미탐 여유. 합성 정답 재현 오차 **4ms**.

### 2-2. 엔진 개선 — 확실히 고친 것 ✅

**① 너무 짧게 자르던 문제 (배포 완료)**
- 첫 측정에서 엔진 **Hit@5 = 0.00**. 원인은 "못 찾아서"가 아니라 정답 구간을 찾고도 **4~8초로 잘라** IoU가 무너진 것.
- 프롬프트 모순("펀치라인만/짧게")을 "셋업 포함 30~60초"로 재작성 + 30초 하한 확장.
- → **확정 성과는 "추천 길이 8초 → 45초"**(구조적, 실측). 당시 Hit@5 0.00→0.67도 나왔으나 이건 단일·소표본(정답 3건)이라 **§3에서 표본 확대 후 ~0.20로 정정**됨 — 0.67을 성과 수치로 읽지 말 것.

**② 대사 중간에서 시작/끊기던 문제 (이번 작업, 배포 완료)**
- 기존엔 경계를 **장면(시각 컷)**에만 맞춰, 말하는 도중 컷이면 "갑자기 대사 시작"·"말 뚝 끊김"이 났다.
- **발화(문장) 타임스탬프로 스냅**: 시작이 대사 중간이면 문장 처음으로, 끝이 대사 중간이면 문장 끝까지 (2.5초 밖이면 손 안 댐). AI·휴리스틱 두 경로 모두 적용, 단위테스트 6/6 통과.

**③ 처리 속도 — 무인 스케일의 전제 (이번 작업, 배포 완료)**
- 실측: 장면감지+프레임 분석(CPU)이 긴 영상 분석 시간의 **최대 74%**(13분 영상이 63분).
- **빠른 모드(`--fast`)**: 대사 기반 콘텐츠는 '터지는 순간'=대사 순간이라, 자막 타임스탬프로 후보
  구간을 만들어 시각 분석을 통째로 건너뛴다. **61분 영상 → 96초**(원정대5), 35분 → 68초(원정대4).
- 전부 additive — 기본은 기존 풀 파이프라인 그대로. 3,000편 규모 처리를 현실화한다.
- 부수 수정: 장면감지 하트비트(긴 영상 stall 오판 종료 방지) + 받아쓰기 degeneration 정제.

### 2-3. 학습 파이프라인 + 자동화 ✅

수동 과정(매칭→설명→학습→적용)을 **채널 버튼 하나로**:
```
매칭(오디오) → 구간 설명(Gemini) → 규칙 학습(LEARN) → 채널 프로파일 저장
  → 이후 그 채널 영상 분석에 자동 주입 (신뢰도 낮으면 가중 약하게 = 과적합 방지)
```
- Lab에서 진행 상황·학습된 규칙(신뢰도·표본·규칙 내용)을 화면으로 확인 가능.
- 독립 확증: LEARN이 뽑은 규칙("60초 초과 회피, 25~60초 최적")이 §2-2 채점 결론과 **같은 방향** → 두 다른 경로가 같은 답.

## 3. 핵심 질문의 현재 답 — "학습이 효과 있나"

### 확실한 것 ✅
- **엔진의 구조적 개선(길이·경계)은 증명됨.** 이건 학습과 무관하게 모든 채널에 유효하다.
- 학습 파이프라인이 **유효한 규칙을 뽑는다**는 것도 확인됨(독립 경로 교차검증).

### 아직 측정 중 ⏳ — 그리고 왜 신중해야 하는가
이번에 **중요한 방법론적 사실**을 발견했다:

> 추천 엔진은 temperature 0인데도 **실행할 때마다 결과가 조금씩 다르다.**
> 같은 조건을 5번 돌리니 Hit@5가 0.00 / 0.33 / 0.33 / 0.67 / 0.67 로 요동쳤다.

즉 **한 번 재서 "학습 켜니 올랐다"고 하면 그게 학습 덕분인지 우연인지 구분할 수 없다.** 그래서:
- **반복 채점기**를 만들어 조건마다 여러 번 반복 → 평균±변동으로 판정하게 했다.
- **홀드아웃을 3편(정답 26건)으로 확대** — 정답이 많아야 우연이 걸러진다.
- **누수 차단**: 학습은 홀드아웃 3편을 제외한 데이터로만 (정답을 미리 본 채로 채점하면 반칙).

### 📊 다중 홀드아웃 A/B 결과 (실측, 2026-07-21)

**설계**: 홀드아웃 3편(원정대5=18정답, 원정대4=5정답, 경주=3정답 · 총 **26 정답**), 누수 없는
프로파일(홀드아웃 제외 22쌍 학습, 신뢰도 0.75, 훅 가중 차등: 돌직구 1.56·반전 1.11·웃음 0.82),
학습 on/off를 **각 5회 반복** 채점해 엔진 변동을 상쇄.

| 지표 | 학습 끔 | 학습 켬 | 상대개선 | 판정(차이 vs 변동σ) |
|---|---|---|---|---|
| Hit@5 | 0.100 | 0.138 | +38% | 효과 있음 (0.038 > 0.031) |
| Hit@10 | 0.131 | 0.200 | +53% | 미확정 (0.069 ≈ 0.071) |
| Hit@20 | 0.154 | 0.215 | +40% | 효과 있음 (0.061 > 0.060) |

**결론 두 개 (섞지 말 것):**

1. **학습은 반영됐고 효과가 있다** ✅ — 모든 지표에서 학습 켬이 일관되게 높다(+38~53%). 3개 중
   2개가 통계적으로 변동을 넘는다. "학습이 픽을 개선하는가"의 답은 숫자로 예스.
2. **하지만 아직 "사람 손 없이" 수준은 아니다** ❌ — 절대 Hit@10 = 0.20. 엔진 top-10이 사람
   발행 숏폼의 ~20%만 회수. 무인 발행은 아직 무리, 사람 검수 필요.

**중요**: §3의 초기 단일 홀드아웃 0.67은 표본(3정답)이 작아 낙관적이었다. 홀드아웃·정답을 26개로
확대하니 현실은 ~20%다. 이게 과장 없는 실측이며, 실사에서 깨지지 않는 숫자다.
(참고: IoU≥0.5 엄격 매칭 + fast 모드(시각신호 없음)가 하방 압력. 그걸로 부풀리지 않음.)

## 4. 정직한 한계

- **엔진 절대 성능이 아직 낮다** — Hit@10 = 0.20. 무인 발행 전에 사람 검수가 필요하다.
- 매칭 72쌍은 **아직 사람이 확인하지 않았다**(오디오 정렬 신뢰도는 높지만 표본 검수 필요).
- 채널 **1개**(하하PD)만으로는 "이 방법이 모든 채널에 통한다"는 일반화가 안 된다.
- 원본 롱폼이 없는 채널(이미 잘린 클립만 올리는 편성)에는 오디오 매칭 자체가 성립하지 않는다.
- fast 모드는 **시각 신호를 안 본다** — 대사 없는 시각 중심 순간엔 약하다(대사 기반 채널엔 충분).

## 5. 다음 (숫자가 정하는 순서)

1. **엔진 절대 성능 끌어올리기** — 지금 Hit@10=0.20. 학습·프롬프트·후보 생성 개선으로 무인 수준(예: 0.5+)에 접근.
2. **매칭 사람 검수** — 자동 매칭 표본 확인 (Lab **영상별 작업 탭**에서 열람 가능).
3. **다운로드 지역차단·레이트리밋** — 현재 실질 병목(미국 VM이 한국 지역잠금 영상 일부 못 받음).
4. **채널 확대** — 다른 채널에도 적용해 일반화 검증.
5. **fast 대량 생성** — 빠른 모드로 롱폼 대량 분석 → "우리가 짜른 클립" 데이터셋 3,000 실현.
6. **풀 파이프라인 필요성 검증** — 같은 홀드아웃에 **fast vs full Hit@N 직접 비교**. 무거운
   단계(장면감지·프레임비전·cast·timeline·narrative)가 추천을 유의미하게 개선하는지 숫자로.
   개선 없으면 → **추천 기본값 = fast, 무거운 분석은 on-demand**(메타데이터 필요할 때만).
   (cast·narrative는 추천보다 에피소드 분석 화면·에디터 맥락용 — "추천엔 과하나 다른 데 쓴다"는 별개 정당화.)

## 6. 다음 연구 방향 — "무엇이 쇼츠를 터지게 하나" (자름 다음은 포장)

지금까지 = **"어디를 잘랐나"**(Hit@N — 사람이 고른 구간을 찾는가). 다음 = **"어떻게 포장했나"** —
같은 구간이라도 조회수를 가르는 것. **고조회 vs 저조회 숏폼을 이 축으로 대조**한다.

**분석 축:**
1. **제목** — 어그로성·궁금증 유발 강도. ⚠️ 제목 표면 키워드만으론 예측 불가 실측(AUC 0.385) →
   키워드가 아니라 "궁금증 갭·낚시"의 **의미**를 봐야 한다.
2. **오버레이** — 화면에 박힌 텍스트·그래픽(상황 설명·강조 자막).
3. **첫 장면(오프닝 훅)** — 첫 1~3초에 뭐가 나오나(얼굴·강한 리액션·텍스트·떡밥).

**가설**: 고조회를 가르는 건 대개 **어그로성 제목 + 강한 첫 장면**.

**필요 데이터(신규)**: 지금 분석 대상은 **롱폼 소스 구간**이다. 포장을 보려면 **발행된 숏폼 자체**를
분석해야 한다 — 제목(보유) + **오프닝 프레임·오버레이 텍스트(비전·OCR 필요)**. 매칭된 숏폼
videoId로 각 숏폼의 첫 프레임·화면 텍스트를 뽑아 성과 tier와 대조 → "포장이 조회수를 가르는가"를
숫자로. (자름 품질 Hit@N과 **분리해서** 측정할 것 — 두 축은 독립이다.)

## 7. 엔진 개선안 — 학습 데이터로 가중치·프롬프트 강화 (우선순위)

지금 학습 효과가 소폭인 원인: ① 재랭킹은 **이미 찾은 후보만** 재정렬(맞는 순간이 후보에 없으면 못 살림),
② 훅 카테고리 신호가 거칠고 더러움(8개·표본 22~40·hook 필드 오염 이력), ③ 추상 규칙은 LLM을 약하게 이끔.

| # | 개선 | 왜 | 비용 |
|---|---|---|---|
| **①** | hook 스키마 enum 강제 + 설명 100% 채우기(40→72) | garbage in→garbage weights. 깨끗해야 lift가 진짜 신호 | 소 |
| **②** | 가중치에 **길이·감정 축 추가** (훅만 X) | 실측상 제일 깨끗한 신호는 길이(고52 vs 저60), 감정도 52/52 깨끗 | 소 |
| **③** | 프로파일을 **후보 생성(phase 1)**에도 세게 | 재랭킹은 늦음 — 맞는 순간이 애초에 후보가 되게 | 중 |
| **④** | 프롬프트에 **실제 고성과 예시 few-shot** (추상 규칙 X) | LLM은 구체 예시를 훨씬 잘 따름. 고성과 구간 자막 이미 보유 | **소·효과 클 듯** |
| **⑤** | 손수 가중치 → **훈련된 스코어러**(`apply_learned_rerank`) | 진짜 학습. 데이터 충분(3,000) 후 LightGBM | 대·후순위 |

**핵심 원칙**: 학습 데이터를 **규칙으로 요약해 넣지 말고 원본 예시·깨끗한 통계로** 넣는다(요약하면 신호가 뭉개짐).
**착수 순서**: ④+① 먼저(코드 조금, 효과 큼) → ③② → ⑤. **각 변경은 fast 홀드아웃 Hit@N으로 측정 후 채택**(무측정 반영 금지).
