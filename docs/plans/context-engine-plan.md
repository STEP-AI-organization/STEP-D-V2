## 루프 전체 그림

```
[캐스트 등록] → [프레임: 인물 확정 + 배경/상황] → [장면 요약] → [시간대 롤업(서사 타임라인)]
                                                                        ↓
[인물 KB 갱신] ←── [배포 성과·댓글 반응(인물별)] ←── [배포] ←── [쇼츠/하이라이트 선정]
      └──────────────── 다음 회차 분석·선정의 컨텍스트로 주입 ────────────────┘
```

## 계층별 설계

### CX-0. 프로그램 메타데이터 인제스트 (온보딩 — 캐스트 등록보다 먼저)

프로그램을 서비스에 붙이는 순간, "인터넷에 '나는솔로' 쳤을 때 나오는 정보"(시놉시스·장르·연령등급·출연자·제작진·방영 정보·포스터)를 우리 DB로 구조화한다. Netflix 프로그램 페이지가 목표 스키마의 좋은 참조: 제목/연도/시리즈 유형/장르/시놉시스/출연/크리에이터.

**스키마 — 4계층 (넷플릭스 오징어게임·참교육 페이지 + 위키백과 나는솔로 실조사 기반, 2026-07-15)**

**경로 A — 기존 방영작: 웹 자동 수집 + 운영자 확정**

- 프로그램명 입력 → LLM 웹 검색으로 위 스키마를 자동 채움 → 운영자가 검수·수정 후 저장 (자동 저장 금지 — 오정보·동명 프로그램 리스크).
- 소스 우선순위: ① 방송사 공식 페이지(고객사 자산이라 권리 문제 없음) ② 위키백과(CC BY-SA, 출처 표기) ③ TMDB API(무료, attribution 필요). 나무위키류는 라이선스·정확성 문제로 참고만.
- 방영 중 프로그램은 회차 추가 시마다 재수집(출연자 변동 — 예능 기수 교체 대응).

**경로 B — 미방영작 (우리를 통해 첫 공개): 사전 등록**

- 인터넷에 정보가 없으므로 방송사가 직접 입력하는 사전 등록 폼 — 편성기획서·보도자료에서 나오는 수준의 필드 (위 스키마와 동일 + 회차별 시놉시스).
- ⚠️ 미방영작 정보는 유출 사고 리스크가 큼: 공개일 전까지 접근 권한 제한(프로그램 단위 권한), 감사 로그, 공개일 도래 시 자동 해제. 계약상 비밀유지 조항과 연동.
- 방영 시작 후에는 경로 A 재수집으로 보강(시청자 반응 기반 키워드 등).

**프로그램 이해 프로파일 — 메타데이터를 넘어 "재미 문법"까지 (선정 품질의 전제)**

프로그램 자체를 이해해야 쇼츠·클립이 잘 뽑힌다. 같은 "말다툼 장면"이라도 나는솔로에서는 최고의 쇼츠감이고 다큐에서는 아니다. 그래서 CX-0은 사실 정보 위에 **프로그램 이해 프로파일**을 한 층 더 만든다:

| 항목 | 예 (나는솔로) | 주입되는 곳 |
|------|--------------|------------|
| 포맷 문법 | "일반인 연애 리얼리티. 기수제. 자기소개→데이트 신청→최종선택 구조" | 장면 이해(CX-2) — "지금이 최종선택 단계"를 알고 요약 |
| 시청 포인트 (이 프로그램은 뭐가 재밌나) | 삼각관계 긴장, 돌직구 발언, 예상 밖 선택, 출연자 캐릭터성 | **비전 평가 루브릭(E)에 주입** — 범용 "재미" 대신 "이 프로그램에서 터지는 장면" 기준으로 평가 |
| 훅 카테고리 가중치 | '반전'·'돌직구' 가중↑, '정보성' 가중↓ | 후보 탐지(C) |
| 대표 밈·코드 | "영숙", "정숙의 결정" 등 시청자 커뮤니티 코드 | 제목 생성(I), 트렌드 매칭(T) |
| 금기·주의 | 스포일러 경계(최종선택 결과), 민감 출연자 이슈 | 선정 필터, 예고 컷 지점 |
| 편집 톤 | 자막 드립 많음, 리액션 컷 위주 | 자막 템플릿·렌더 프리셋(F) |

- 생성: 경로 A는 시놉시스+리뷰·커뮤니티 반응을 LLM으로 요약해 초안 생성, 경로 B는 방송사 기획의도서 기반 — 둘 다 **운영자 검수 후 확정**, 이후 성과 환류(CX-5)로 갱신.
- 이 프로파일이 발명신고서 구성 E의 "4축 루브릭"을 프로그램별로 특화시키는 장치다: 평가 프롬프트가 회차마다 동일한 범용 기준이 아니라 프로그램 이해를 탑재한 기준이 된다.

**효과** — CX-1 캐스트 등록이 "빈손 시작"이 아니라 출연자 이름+프로필 사진이 미리 채워진 상태에서 시작하고, 첫 회차 분석부터 시놉시스·톤·재미 문법이 프롬프트에 들어가 장면 이해와 선정 품질이 초기부터 확보된다. 미방영작의 경우 우리가 첫 배포 전에 이미 완전한 컨텍스트를 갖는 것 — 늦게 붙는 경쟁 도구와의 구조적 격차.

### CX-1. 캐스트 레지스트리 (인물 등록)

> **[진척 2026-07-16]** 현재 코드의 캐스트는 프로그램 생성 폼의 쉼표 구분 free-text 문자열 배열이 전부다
> (`apps/web/src/components/new-program-dialog.tsx`의 `cast` 입력 → `POST /api/programs`가 `cast: string[]`로
> entities에 저장, `apps/web/src/lib/types.ts`의 `Program.cast?: string[]`). 역할·기수 구분, 대표 얼굴 이미지,
> 얼굴 매칭, 미지 얼굴 클러스터링은 전부 미구현 — 이 free-text cast가 아래 레지스트리의 시드가 된다.

- 프로그램·기수(시즌) 단위로 출연자 등록: 이름/역할(출연자·MC)/대표 얼굴 이미지 수 장.
- 방송사는 이 정보를 이미 갖고 있음 → 운영자 입력 UI + 1화 분석 후 "미등록 얼굴 클러스터"를 보여주고 이름 붙이는 반자동 온보딩.

**동적 캐스트 발견 — 레지스트리는 고정이 아니라 회차마다 갱신된다 (핵심)**

일반인 출연 예능은 웹 시놉시스에 출연자 정보가 없거나 부실하고(CX-0 경로 A가 못 채움), 방영 중 신규 투입(속칭 "메기") · 게스트 · 스핀오프 합류로 인물이 계속 늘어난다. 따라서 매 회차 분석에 **미지 인물 발견 루프**를 내장한다:

1. **미지 얼굴 클러스터링**: 회차 분석 중 등록 캐스트와 매칭 실패한 얼굴들을 임베딩 유사도로 클러스터링 → "동일 인물로 보이는 미등록 얼굴 N개 샷" 단위로 묶음.
2. **이름 자동 제안 — 방송 신호 활용**: 새 인물 등장 시 방송이 스스로 정보를 준다.
   - 네임 자막 OCR: 첫 등장 때 이름·나이 자막이 번인되는 관행 (일반인 예능 대부분) —
     **[진척 2026-07-16]** `core/names.py`가 이 앵커 가설을 Gemini OCR로 실측하는 검증 실험으로 이미 존재
     (장면별 이름 자막 커버리지·등장 이름 집계. 인물 확정 스택 본체는 아님)
   - 호명 탐지: STT 전사에서 다른 출연자가 부르는 이름 ("현숙 씨는 어떻게 생각해요?")
   - 등장 통계: 샷 수·등장 시간·상호작용 인물
3. **운영자 확정 큐**: "새 인물 감지: '현숙' 추정(자막 OCR 신뢰도 높음) · 14개 샷 · 22분 등장 · 주 상호작용: 상철" → 승인/이름 수정/무시(스태프·행인). 승인 시 캐스트 등록 + **해당 회차 소급 재태깅**.
4. **역할 자동 분류**: 등장 회차 수·시간으로 고정(다회차)/게스트(1회)/신규 투입(중반 합류) 구분 — 인물 KB와 선정 가중치에 반영 (신규 투입 인물은 화제성 부스트 후보: "메기 등장" 자체가 쇼츠 소재).
5. **무시 목록**: 스태프·제작진·반복 행인은 "무시" 처리하면 이후 회차에서 다시 묻지 않음 (임베딩 유지).

이 루프 덕에 CX-0에서 출연자 정보를 못 채운 프로그램도 1~2화 분석이면 캐스트가 자동 구축되고, 시즌 중반 투입 인물도 등장 회차에 즉시 포착된다. — 웹 정보에 의존하는 경쟁 도구가 흉내 낼 수 없는 부분.
- 얼굴 식별 기술 (상세 조사·다중 신호 융합 표·PoC 게이트는 [../research/object-detection-research.md](../research/object-detection-research.md)의 "CX 인물 확정 식별 스택" 참조):
  - **1차 (관리형): Gemini 멀티모달** — 등록된 대표 얼굴들을 레퍼런스로 프롬프트에 넣고 프레임 속 인물 매칭. 출연자 15명 수준의 닫힌 후보군이라 정확도 유리 (오픈셋이 아니라 "이 15명 중 누구" 문제).
  - **2차 (물량 증가 시): 얼굴 임베딩 + pgvector**(Cloud SQL 확장) 검색으로 전환. ⚠️ InsightFace 등 유명 모델은 코드 MIT여도 **모델 가중치가 비상업**인 경우가 많음 — 도입 시점에 라이선스 재조사 필수.
- ⚠️ 법무: 얼굴 인식은 개인정보(민감정보) 처리 — 방송사와의 계약에 출연자 초상 데이터 처리 근거 명시 필요. B2B라 방송사가 출연 계약상 권리를 보유하는 구조를 활용.

### CX-2. 장면 이해 (프레임 → 장면 → 시간대 롤업)

- 샷 분할(ffmpeg 씬 검출) → 샷당 대표 프레임 1~2장만 분석 (프레임 전수 분석 금지 — 비용 상한).
- 샷 단위 Gemini 호출에 **3중 컨텍스트 주입**이 핵심:
  1. 확정 인물 ID (CX-1 결과: "영숙, 영호")
  2. 해당 구간 전사 텍스트 (B의 transcript — 화자 분리를 얼굴과 교차해 "누가 말했는지"까지 귀속)
  3. 인물 KB 요약 (지금까지의 서사: "영숙은 3화에서 영호에게 호감 표시")
  → 출력: 구조화된 장면 레코드 `{시각, 공간, 등장인물, 행동, 요약문("영숙이 영호에게 노래를 불러주고 산책 중")}`
- 롤업: 장면 레코드들을 시퀀스(수 분)→회차 단위로 LLM 요약 → **서사 타임라인** 테이블. 회차 전체 시놉시스가 부산물로 나옴 (메타데이터 I에도 재사용).

### CX-3. 인물 지식 베이스 (KB)

인물별로 회차·배포가 진행될 때마다 갱신되는 누적 데이터:

| 필드 | 원천 | 갱신 시점 |
|------|------|----------|
| 기본 프로필 (이름·기수·역할) | 운영자 등록 | 온보딩 |
| 회차별 행동 로그 | CX-2 장면 레코드 | 회차 분석 완료 시 |
| 페르소나/이미지 태그 ("직진남", "밈 캐릭터") | 장면 로그 + 댓글에서 LLM 추출 | 회차·배포 후 |
| 시청자 반응 (인물별 댓글 언급량·감성·기대 포인트) | 배포 클립 댓글 분석 (J 확장 — 댓글 요약을 인물 단위로 구조화) | 배포 후 지속 |
| 인물별 성과 (그 인물 중심 쇼츠의 조회·완주율) | 채널 분석 파이프라인 metrics × 클립 등장인물 태그 | 배포 후 지속 |

- 저장: `person`, `person_episode_log`, `person_reaction` 정규 테이블 (신설 필요). 클립에는 등장인물 태그가 붙음 (성과 귀속의 키).

> **[진척 2026-07-16]** person 계열 정규 테이블은 아직 존재하지 않는다 — `apps/server/schema.sql`은
> entities/media/kv + YouTube 수집 6종(youtube_channels·channel_videos·video_stats·video_analytics·
> video_retention·video_comments)의 9개 테이블뿐이고, 런타임 CREATE TABLE 코드(queue.ts·db-pg.ts)에도
> person 테이블은 없다. 클립 등장인물 태그도 미구현.

### CX-4. 선정 통합 (쇼츠/하이라이트 결정)

계획된 점수 구조([pipeline-plan.md](pipeline-plan.md) 구성 E)에 서사·인물 축 추가:

```
최종점수(후보, 채널) = [ 비전 0.70 + 로컬(훅) 0.30 ]           ← 계획 (pipeline-plan.md E, 미구현)
                      × 채널 적합 계수                          ← 계획 (pipeline-plan.md E, 미구현)
                      × 인물·서사 계수 (신규 — 이 문서)
```

> **[진척 2026-07-16]** 위 두 줄은 "이미 있는 베이스"가 아니라 pipeline-plan.md의 미구현 계획이다.
> 현행 쇼츠 선정(`core/recommend.py`)은 vision_score가 포함된 장면 타임라인을 **단일 Gemini 추론 호출**
> 하나로 처리하며, 0.70/0.30 수치 가중 융합도 채널 적합 계수 곱셈도 코드에 없다.
> 인물·서사 계수는 그 융합 구조가 구현된 뒤에 얹는 3번째 축이다.

인물·서사 계수의 입력:
- **인물 화제성**: 지난 배포에서 이 인물 쇼츠의 성과 배수 + 댓글 언급량 추이
- **기대 매칭**: 댓글에서 추출한 기대 포인트("영숙 답장 언제 나옴?")와 이번 후보 장면의 매칭 — 시청자가 기다리던 장면은 강한 부스트
- **서사 완결성**: 장면 레코드 기준으로 후보 구간이 행동의 시작~끝을 포함하는지 (문장 단위 스냅 D를 장면 단위로 확장)
- 하이라이트(롱폼)와 쇼츠는 같은 서사 타임라인에서 길이·완결성 기준만 달리해 선정.

### CX-5. 환류

- 배포 → 인물별 성과·댓글이 KB 갱신 → 다음 회차의 CX-2 프롬프트 컨텍스트와 CX-4 계수에 자동 반영.
- 채널 분석 파이프라인(pipeline-plan.md)의 (프로그램, 채널, 훅, 길이) 차원에 **인물** 차원 추가 — "이 채널에서는 영숙 클립이 평균 2.3배" 같은 단위로 축적.

## 발명신고서와의 관계 (특허 확장 후보)

구성 B(단일 원천 공유)·E(융합 평가)·J(성과 환류)의 확장이며, 다음이 신규 청구 후보로 보임:
1. 사전 등록 캐스트 기반 폐쇄 후보군 인물 확정 식별 + 화자 분리 교차에 의한 발화 귀속
2. 인물 KB(서사·반응·성과)의 회차 간 누적과 이를 선정 계수로 환류하는 폐루프
3. 댓글 기대 포인트와 미방영 장면 매칭에 의한 선정 부스트
4. 미지 얼굴 클러스터링 + 네임 자막 OCR + 발화 호명 탐지의 교차에 의한 신규 출연자 자동 발견·등록 및 소급 재태깅 (동적 캐스트 발견 루프)
→ 변리사 검토 목록에 추가 권장.

## 마일스톤 (CX 트랙 — AP 트랙과 병행)

- **CX0** (AP2와 병행 가능 — 가장 저비용): 프로그램 온보딩 폼 + 웹 자동 수집(경로 A) + 프로그램 이해 프로파일 초안 생성. 미방영작 사전 등록(경로 B)은 수요 발생 시.
- **CX1** (AP2 이후): 캐스트 등록 UI + 샷 분할 + Gemini 인물 확정 식별. CX0의 출연자·프로필 사진을 시드로 사용. 클립 등장인물 태그부터 시작 (성과 귀속의 최소 단위).
- **CX2** (AP3와 병행): 장면 레코드 + 서사 타임라인. 회차 시놉시스 자동 생성.
- **CX3** (AP4 이후): 인물 KB + 댓글 인물별 구조화. 
- **CX4** (AP5): 인물·서사 계수 선정 통합 + 기대 매칭 부스트 + 인물 차원 성과 환류.

## 비용 통제

- 장면 이해는 샷당 1~2프레임 + 프록시 해상도 + 배치 호출. 90분 회차 ≈ 샷 수백~1천 개 수준 → 회차당 호출 상한을 config로 고정 (발명신고서의 비용-계단 원칙 유지).
- 인물 식별은 "등장 샷"에만 (씬 검출 + 얼굴 존재 프레임 필터 후 호출).
- KB 요약은 프롬프트에 전문이 아닌 압축 요약본만 주입 (토큰 상한 고정).
