## 0. 한 문단 요약

채널이 **실제 발행한 숏폼**을 정답지로 삼아 추천 엔진을 채점하는 체계를 세웠다. 첫 홀드아웃
측정에서 현행 엔진의 **Hit@5 = 0.00**(추천을 4~8초로 너무 짧게 자름)이 드러났고, 프롬프트·길이
하한을 고쳐 **Hit@5 = 0.67**로 올렸다. 완전히 독립적인 경로(발행 성과 데이터의 LLM 규칙 학습)가
같은 결론("60초 초과 회피 / 25~60초 최적")을 내놓아 개선을 교차 검증했다. 종결어미 스냅은
효과가 실측되지 않아 폐기했다 — 평가 체계가 "좋아 보이는 아이디어"를 숫자로 걸러낸 사례다.
**결론: 채널 데이터로 엔진을 강화할 수 있음을 방법론적으로 실증했다. 단, 홀드아웃 1편·표본
소규모이므로 통계적 확정은 추가 측정 후.**

## 1. 실험 인프라 — 무엇을 만들었나 (재현 가능)

| 산출물 | 역할 |
|---|---|
| `core/align.py` | 숏폼↔롱폼 **오디오 상호상관 매칭** — Gemini 없이 CPU로 구간 역추적 (임계 score≥0.80·ratio≥1.25, 실측 보정) |
| `core/segment.py` | 매칭 구간의 자막·장면요약·감정·훅을 **Gemini 1회**로 생성 (롱폼 전체 아닌 구간만) |
| `core/evaluate.py` | 발행 숏폼을 정답으로 **Hit@N·경계오차** 채점 (IoU≥0.5) |
| `core/eval_batch.py` | 여러 홀드아웃 **micro-average 집계** (단일 영상 우연 배제) |
| `core/point_profile.py` | 고성과 vs 저성과 **통계 대조** (hook·emotion·길이 lift) |
| `core/learn_profile.py` | 통계+실텍스트 → **채널 규칙**(LEARN) → recommend 프로파일 변환 |
| Lab 매칭·데이터셋 UI + `match.align`/`match.segment`/`match.learn` 잡 | 채널 단위 **버튼 하나로** 매칭→설명→학습 자동화 |

핵심 원칙: **성과는 절대 조회수 금지**. `/api/lab/match/export`가 ±90일 동시기 채널 중앙값 대비
배수(tier)로 정규화 — 3년 전 50만과 지난달 20만을 같은 잣대로 비교하는 오류를 구조적으로 차단.

| 항목 | 값 |
|---|---|
| 자동 매칭 쌍 | **71쌍** (오디오 정렬, 사람 확인 전) |
| 소스 롱폼 | **33편** |
| 성과 tier 분포 | high 26 · mid 24 · low 21 (**균형** — 학습에 이상적) |

## 3. 핵심 실험 — 엔진 개선 전/후 (홀드아웃 채점)

**방법**: 하하PD "경주 PC방"(13분) 롱폼을 분석 → 추천 생성 → 그 채널이 실제 발행한 숏폼 3건을
정답으로 채점. 홀드아웃 롱폼은 학습에서 제외(누수 차단).

| 지표 | BEFORE (현행) | AFTER (개선) |
|---|---|---|
| **Hit@5 (IoU≥0.5)** | **0.00** | **0.67** |
| 정답 탐지 | 0 / 3 | **2 / 3** |
| 추천 길이 | 4~8초 | 45~48초 |

### 진단 (숫자가 원인을 지목)
- BEFORE 0.00은 "엔진이 못 찾은" 게 아니다. 정답 3건 중 1건은 **같은 순간을 찾고도**(키오스크
  구간) 8초로 잘라 IoU 0.24로 미달했다. **문제는 "어디"가 아니라 "얼마나 길게"였다.**
- 원인: 프롬프트 모순. "빌드업 넣지 마라/펀치라인만/짧고 강하게"가 "30~60초" 지시를 덮었다.
  실제 편집자의 40초 클립은 **셋업→터짐→여운**을 담는데, 프롬프트가 그걸 금지하고 있었다.

### 개선 (2건)
1. **길이**: 프롬프트를 "셋업 포함 30~60초 기본"으로 재작성 + 검증에 30초 하한(짧으면 장면
   경계로 앞뒤 확장). → 로그에서 6→48초, 14→45초로 교정 확인.
2. **커버리지**: 추천 수 `n`을 영상 길이 비례로(10분당 3개, 상한 20). 13분에서 편집자는 3개를
   뽑는데 엔진은 5개 고정이라 후반부를 놓쳤다.

**프로덕션 배포 완료** — 이후 분석되는 모든 영상에 적용.

## 4. 교차 검증 — 독립 경로가 같은 결론

개선(§3)은 **홀드아웃 채점**에서 나왔다. 완전히 다른 경로 — **발행 성과 데이터의 LLM 규칙
학습**(LEARN, `learn_profile.py`) — 을 같은 채널에 돌린 결과:

> **고성과 규칙**: "명확한 단호한 주장·고정관념 파괴·빠른 클라이맥스 직행"
> **회피 규칙**: "**지나치게 긴 서사·배경 설명으로 핵심까지 오래 걸림(특히 60초 초과)**"
> **최적 길이: 25~60초** (confidence 0.6)

→ 홀드아웃 채점이 "8초→45초로 늘려라"라고 했고, 발행 성과 데이터가 독립적으로 "60초 초과 회피,
25~60초 최적"이라고 했다. **두 증거가 같은 방향을 가리켜 신뢰도가 상승**했다. 사용자 제안
프롬프트 A(LEARN)가 실제로 유효한 규칙을 뽑는다는 것도 확인됐다.

## 5. 폐기한 실험 — 정직성의 증거

**종결어미 스냅**(한국어 문장 끝에 클립 경계 스냅, 계획서 P1로 우선순위였던 항목):
- 만들고 검증했으나 홀드아웃에서 Hit·경계오차 **모두 변화 없음**(11.4초 유지).
- 원인: 남은 경계오차는 "문장 중간 컷"이 아니라 "엔진과 편집자의 클립 범위 차이"였다. ±3초
  미세조정으로는 11초 차이를 못 메운다. 오히려 멀쩡한 경계를 애매하게 옮기기도 했다.
- **프로덕션에 넣지 않고 폐기.** 효과 없는 코드는 리스크만 는다.

→ 이건 실패가 아니라 **평가 체계가 제 역할을 한 것**이다. "좋아 보이는 아이디어"를 숫자로
걸러냈다. 계획서가 P1로 지목했어도 데이터가 아니라고 하면 데이터를 따른다.

## 6. 자동화 — "채널 분석 한 번 = 학습 + 적용"

수동 과정(export→learn→주입)을 버튼 하나로:
```
POST /api/lab/match/learn (채널)
  → match.learn 잡: 미설명 구간 있으면 match.segment 먼저 자동 실행·재시도 →
     export 구성 → learn_profile(Gemini 규칙) → youtube_channels.pointProfile 저장
  → 이후 그 채널 영상 분석: content-pipeline이 episode.sourceChannelId로 프로파일 찾아
     recommend에 --profile 자동 주입 (기존 스티어링 배선 재사용)
```
- 계획서가 지적한 **"채널→에피소드 연결 부재"를 메움**(from-youtube가 videoId→channelId 저장).
- confidence 낮으면 hookWeights 가중 약하게(1+0.5·conf) — **소표본 과적합 자동 방지**.
- 실측 확인: 트리거 시 잡이 "미설명 51건 → 먼저 채우고 재시도"로 스스로 선행조건을 판단.

## 7. 결론 (과장 없이)

| 주장 | 근거 | 상태 |
|---|---|---|
| 롱폼→숏폼 전환 시 조회수 평균 9.28배 | 실측 | ✅ 강함 |
| 제목 표면 특징만으론 성과 예측 불가 | AUC 0.385(무작위 이하) | ✅ 강함 |
| **엔진 개선으로 편집자 구간 재현율 0%→67%** | 홀드아웃 채점 | ✅ **직접 증거**(단 1편) |
| 발행 성과 데이터가 개선을 교차 검증 | LEARN 규칙 = 채점 결론 | ✅ 독립 확증 |
| 자동 매칭 파이프라인으로 학습 데이터 재현 | 71쌍, 실패 0 | ✅ 실증 |
| 채널 학습이 통계적으로 성과 향상 | — | ⏳ **미확정**(홀드아웃 확대 필요) |

**핵심 결론**: 채널 데이터로 쇼츠 추천 엔진을 강화할 수 있음을 **재현 가능한 방법론으로
실증**했다. Hit율 0→67%는 직접 증거이고, 독립 경로(성과 데이터 학습)가 이를 확증한다. 이는
48건 단발 상관분석(p=0.097)보다 강한 근거다 — 제품의 실제 목적(편집자가 고른 구간을 찾는가)에
직결되고, 재현 가능하기 때문.

**정직한 한계**: 홀드아웃 1편·정답 3건·매칭 71쌍(사람 확인 전)·구간 설명 20건. "통계적으로
입증됐다"는 아직 못 쓴다. 원정대5(정답 18건) 등 홀드아웃 2~3편을 더 채점하면 확정할 수 있다 —
그 파이프라인은 이미 만들어져 있고, YouTube 다운로드 레이트리밋 쿨다운만 풀리면 재개된다.

## 9. 앞으로 더해야 할 것 (전체 로드맵)

§8은 당장의 재개 작업이다. 여기는 "실증"에서 "제품"으로 가기 위해 채워야 할 큰 구멍들을,
**주장의 성패를 좌우하는 순서**로 정리한다.

### 9-1. 가장 중요 — "채널 학습이 성과를 올린다"는 아직 미증명이다 ⭐
§3의 0→67%는 **일반적 엔진 수정(길이·커버리지)**이 효과 있음을 증명했다. 이건 어느 채널에나
적용되는 개선이지, **"이 채널만의 학습된 규칙"이 효과 있다는 증명이 아니다.** 지금 상태로
"채널 데이터로 강화"를 주장하면 근거와 주장이 어긋난다. 닫아야 할 실험:

- **learn → apply → measure 폐루프**: 같은 홀드아웃을 (a) 채널 프로파일 **없이** vs (b) 학습
  프로파일 **적용** 두 번 추천 생성 → Hit@N·고성과 순위를 비교. 배선(`content-pipeline`이
  `episode.sourceChannelId`로 프로파일 주입)은 이미 있으니, 프로파일 on/off 플래그만 있으면 된다.
- **누수 차단**: 홀드아웃 롱폼의 매칭은 학습에서 반드시 제외(영상 단위 hold-out). 지금 learn은
  채널 전체를 쓰므로, 홀드아웃 롱폼을 빼는 필터가 필요하다(신규).
- 이 실험이 (b)>(a)를 보여야 §7 표의 마지막 줄(⏳ 미확정)이 ✅로 바뀐다. **이게 최우선.**

**9-1 실측 (2026-07-21) — 답: 지금 방법으론 판정 불가.** learn→apply A/B를 홀드아웃(경주 PC방,
정답 3건)으로 돌렸다. 그런데 **같은 조건(무프로파일)을 5회 반복하니 Hit@5가 0.00·0.33·0.33·
0.67·0.67로 요동쳤다**(평균 0.40, 범위 0.00~0.67). Gemini가 temperature 0인데도 추천이 실행마다
달라진다. 즉 프로파일 on/off 차이(±0.33)가 **엔진 자체 변동 범위 안에 완전히 묻힌다.** 단일
홀드아웃·정답 3건으로는 채널 학습의 효과를 통계적으로 분리할 수 없다.
→ **필요조건이 둘로 늘었다**: (1) 홀드아웃·정답 수 확대(영상 여러 편·정답 수십 건), (2) 조건마다
**다회 반복 평균**(변동 상쇄). 그 전엔 A/B 한 번의 숫자를 근거로 쓰면 안 된다.
→ 부산물로 안전장치 2개를 넣었다: ① content-pipeline **신뢰도 게이트**(conf<0.75 프로파일은
자동 적용 안 함 — 검증 안 된 프로파일이 회귀를 내는 걸 차단), ② hookWeights를 **통계적 lift로
차등**(균일 가중이 랭킹만 흔드는 문제 수정). 둘 다 "효과 확인 후 켠다"는 원칙의 코드화다.

> ⚠️ **§3 재해석**: 이 변동 발견은 §3의 "0.00→0.67"도 단일 측정이라 재검증이 필요함을 뜻한다.
> 다만 §3에서 **확실한 건 "4~8초로 자르던 구조적 문제"**다(변동과 무관, 프롬프트·하한으로 교정
> 확인). Hit 수치 자체보다 "길이 교정"이 §3의 굳은 성과다.

### 9-2. 정답지 신뢰 — 자동 매칭 71쌍은 사람이 확인한 적 없다
전체 채점이 이 71쌍을 "정답"으로 쓰는데, 전부 `source='auto'`·`confirmed=0`이다. 임계값이
보수적이라 오탐은 적겠지만, **바우처·IR에서 "정답지"라 부르려면 사람이 표본을 검증**해야 한다.
- Lab 매칭 탭에 이미 확정(confirm) UI가 있다 → **무작위 20쌍을 사람이 확인**하고 auto-match
  정밀도(맞은 비율)를 리포트에 명시. "자동 매칭 정밀도 N%"는 그 자체로 강한 실증 수치다.
- 틀린 매칭이 나오면 임계값(score≥0.80) 재보정 근거가 된다.

### 9-3. 평가 지표의 구조적 한계 — 발행 숏폼은 "좋은 순간의 부분집합"
편집자는 좋은 순간을 **다 발행하지 않는다**(경주 PC방에서 3개만 골랐다). 그래서 엔진이 놓친
위치(커플석 436s)가 "실패"가 아니라 "편집자가 안 고른 것"일 수 있다. 현재 recall은 부분 정답
대비라 **과소평가**된다. 보완:
- **정밀도 축 추가**: 엔진 top-N 중 "발행되진 않았지만 합당한" 구간 비율을 사람이 평가(소표본).
  Hit@N(재현)과 함께 봐야 "잘 뽑는가"의 양면이 잡힌다.
- 또는 **고신뢰 음성만 페널티**: 편집자가 명시적으로 버린 구간이 있으면 그것만 오답 처리.

### 9-4. 학습 프로파일 고도화 — 지금은 훅·길이뿐
`learn_profile`이 뽑는 규칙은 hookWeights·targetLength·taboos에 그친다. §3 진단이 지목한 진짜
차이는 **"클립 범위"(어디를 얼마나)**인데, 그건 아직 프로파일에 없다. 더할 신호:
- **채널별 길이 분포**(단일 값 아닌 분포) → validate의 목표 길이를 채널맞춤으로.
- **선호 "순간 유형"**(사건/주장 vs 실황/설명 — LEARN이 이미 언어로 뽑음) → 후보 선정 프롬프트에.
- `recommend`의 `apply_learned_rerank`(예약된 no-op) 자리에 이 신호들을 정량 가중으로.

### 9-5. 일반화 — 채널 1개로는 "도메인 적응"이 아니다
하하PD 하나로는 "이 채널에 맞췄다"까지다. **"채널마다 다른 규칙을 학습한다"**를 보이려면
원본 롱폼이 있는 채널 2~3개가 필요하다. 그런데:
- ENA(클립을 롱폼으로 올림)·드나드나(한국 지역제한)는 매칭이 안 된다(§2 한계).
- → **원본 롱폼이 있는 채널을 추가 확보**하거나, 클립-소스 채널은 다른 방식(제목/설명 파싱)으로
  다뤄야 한다. 최소 2번째 채널에서 "다른 규칙이 나오고 그 채널 홀드아웃에서 효과"를 보여야
  일반화 주장이 선다.

### 9-6. 운영 견고성 — 레이트리밋이 지금 실질 병목
쿠키·nightly로 봇차단은 풀었지만, 대량 백필이 IP+계정을 다시 레이트리밋에 걸었다. 재발 방지:
- match.align/segment 백필 **동시성 1 + 요청 간 지연**(현재 delayMs는 잡 간 시차일 뿐, 잡 내부
  연속 다운로드가 몰린다). 잡 안에서도 다운로드 사이 sleep·지수 백오프.
- 실패가 봇차단이면 **자동 장기 쿨다운**(재시도 폭주가 상황을 악화시킴 — 실제로 겪음).
- 장기적으로 쿠키 회전(만료 대비)·다중 IP.

### 9-7. 폐루프 자동화 — 새 숏폼이 계속 나온다
채널은 계속 발행한다. 학습은 지금 수동 트리거다.
- **주기적 재학습**: 신규 매칭이 N건 쌓이면 자동으로 match.learn 재큐(스케줄).
- **진짜 폐루프**(§channel-domain-adaptation 3단계): 우리가 발행한 클립의 실제 성과를 라벨로
  편입 — 단 실업로드(`YOUTUBE_UPLOAD_ENABLED`)가 켜져야 하고 지금은 off.
