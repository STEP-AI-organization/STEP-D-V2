## 1. 트랙 A — 리텐션 커브 활용 (7-1단계)

**목표**: 리텐션 커브를 구간 피처와 조인 → 정밀 라벨로 전환

### 1-1. 현재 상태
- `youtube.ts:fetchVideoAnalytics`가 `video_retention.curve` JSONB에 저장 중
- 단, **롱폼 원본이 유튜브에 올라와 있고**(하하PD 채널), **OAuth 토큰으로 읽을 수 있어야** 작동
- 하하PD는 조건 충족 — 리텐션 커브는 이미 DB에 쌓이고 있을 가능성 높음

### 1-2. 할 일

| # | 작업 | 상세 | 코드 위치 |
|---|------|------|-----------|
| A-1 | 하하PD 롱폼 리텐션 커브 현황 확인 | `video_retention` 테이블 조회 — 롱폼 33편 중 몇 편이 커브 보유? | `video_retention.curve` |
| A-2 | 구간 리텐션 집계기 | `content_analysis.data.scenes[]`의 각 구간에 대해 커브 매핑 → 구간별 `audienceWatchRatio` 평균/최저/기울기 산출 | 신규: `core/aggregate_retention.py` |
| A-3 | 리텐션 라벨화 | "이 구간은 시청 유지율 80% 유지 vs 50%로 급락" → 구간 라벨로 변환 | A-2에 통합 |
| A-4 | 리텐션 라벨 vs 채택 라벨 일치도 검증 | 운영자 채택 구간이 리텐션 상위 구간과 일치하는가? (엔진 검증의 또 다른 축) | `core/evaluate.py` 확장 |

### 1-3. 성공 기준
- 하하PD 롱폼 중 **50% 이상**이 리텐션 커브 보유 확인
- 구간 리텐션 집계기 v0 완성 (1,000구간 처리 < 10초)
- "리텐션 상위 20% 구간"과 "운영자 채택 구간"의 IoU ≥ 0.3

## 2. 트랙 B — 폐루프 구축 (publish 실제 구현)

**목표**: STEP D 클립을 하하PD 채널에 실제 게시 → 성과 자동 회수

### 2-1. 현재 상태
- `POST /api/distributions/publish`는 **스텁** — 상태 기록만, 실제 송출 없음
- `PATCH /api/clips/:id/link-video`는 **수동** — 사람이 videoId 입력
- YouTube OAuth는 `mode=analytics`만 동작, `mode=publish`는 미구현

### 2-2. 할 일

| # | 작업 | 상세 | 의존 |
|---|------|------|------|
| B-1 | YouTube OAuth publish scope 추가 | 현재 `yt-analytics.readonly` → `youtube.upload` scope 추가, 재동의 | Google OAuth 심사 (Testing 상태면 7일 토큰 수명) |
| B-2 | publish 실제 구현 | `POST /api/distributions/publish` — YouTube Data API `videos.insert`로 클립 영상 업로드 + 메타데이터 설정 | B-1 |
| B-3 | link-video 자동화 | publish 성공 시 `publishedVideoId` 자동 저장 (현행 수동 제거) | B-2 |
| B-4 | 게시 성과 자동 수집 | 게시된 클립의 조회수·리텐션·댓글을 `video.analyze` 잡 경로로 자동 수집 | B-3, `worker.ts` fan-out |

### 2-3. 성공 기준
- 하하PD 채널에 STEP D 클립 **1편** 실제 게시 + 성과 수집까지 폐루프 완주
- "publish" 버튼 클릭 → 게시 → 성과 확인까지 **자동화**

## 3. 트랙 C — 엔진 개선 (성과보고서 §7, 즉시 효과)

**목표**: 코드 몇 줄로 Hit@10을 0.20 → 0.3+로 올린다

### 3-1. C-①: hook 스키마 enum 강제 (소)

| 항목 | 내용 |
|------|------|
| 문제 | 8개 hook 카테고리지만 DB에 오염 이력(공백·소문자·신규값)으로 가중치 신뢰도 저하 |
| 작업 | `recommend.py` 출력 시 hook을 enum 집합(반전/웃음/돌직구/갈등/감동/충격/정보/예상치못한전개)으로 강제 매핑 |
| 설명 100% 채우기 | 현재 40/72건만 설명 있음 → Gemini 프롬프트 강화로 커버리지 100% |
| 효과 | 가중치가 깨끗해져야 lift가 진짜 신호 |

### 3-2. C-②: 가중치에 길이·감정 축 추가 (소)

| 항목 | 내용 |
|------|------|
| 근거 | 실측상 제일 깨끗한 신호는 길이(고성과 52초 vs 저성과 60초), 감정도 52/52 깨끗 |
| 작업 | `profile.hookWeights`에 `lengthFit`·`emotionWeight` 축 추가 |
| 효과 | 단일 훅 가중치보다 표현력 ↑ |

### 3-3. C-③: 프로파일 후보 생성에도 적용 (중)

| 항목 | 내용 |
|------|------|
| 문제 | 현재 프로파일은 재랭킹(phase 2)에만 적용 — 맞는 순간이 후보에 없으면 재랭킹 무용 |
| 작업 | `recommend.py` phase 1(`_extract_candidates`)에도 프로파일 규칙 전달 |
| 효과 | 후보 풀 자체가 채널 맞춤화 |

### 3-4. C-④: 프롬프트에 고성과 예시 few-shot (소·효과 큼)

| 항목 | 내용 |
|------|------|
| 문제 | 지금 프롬프트는 추상 규칙("셋업 포함 30~60초, 펀치라인 중심") — LLM은 구체 예시를 훨씬 잘 따름 |
| 작업 | 하하PD 고성과 구간의 자막/타임라인을 예시로 프롬프트에 주입 (채널별로 다른 예시) |
| 효과 | +38% → +50%+ 목표, 가장 적은 비용으로 큰 개선 기대 |

### 3-5. C-⑤: LightGBM 스코어러 (대·후순위)

| 항목 | 내용 |
|------|------|
| 작업 | `apply_learned_rerank` 구현 — 채택확률 이진분류 → `final_score = appeal × program_fit × learned` |
| 필요 데이터 | 3,000클립 규모. 연구 설계는 `highlight-model-feasibility.md §5` 완료 |
| 착수 시점 | 데이터셋 1,000+ 쌍 이후 |

## 4. 트랙 D — 데이터셋 규모 확장

**목표**: 71쌍 → 1,000+ 쌍으로 확장해 통계 검정력·학습 데이터 확보

### 4-1. 하하PD 전 카탈로그 ingest

| 항목 | 값 |
|------|-----|
| 현재 | 33편 분석, 71쌍 매칭 |
| 하하PD 전체 | 769개 영상 (롱폼 예상 200~300편) |
| 방법 | `--fast` 모드로 큐 밀어넣기 (61분 영상→96초, 300편 ≈ 8시간) |
| 산출 | 예상 500~700쌍 매칭 |

### 4-2. 할 일

| # | 작업 | 상세 |
|---|------|------|
| D-1 | 하하PD 채널 영상 목록 추출 | `youtube.ts` list API로 채널 전체 영상 ID 획득 |
| D-2 | 롱폼만 필터 | duration ≥ 120초 필터 (숏폼 제외) |
| D-3 | fast 분석 배치 | 큐에 `content.analyze` 잡 밀어넣기 (워커 VM에서 순차 처리) |
| D-4 | 매칭 자동 실행 | `align.py` 배치로 모든 신규 분석 ↔ 기존 발행 숏폼 크로스 매칭 |
| D-5 | 구간 설명 일괄 생성 | 매칭 성공 쌍에 대해 `segment.py` 배치 (Gemini 1회/쌍) |

### 4-3. 성공 기준
- 분석 완료 200+ 롱폼, 매칭 500+ 쌍
- 데이터셋 CSV 3종 갱신 (`01_published_catalog_labeled.csv` 등)

## 5. 트랙 E — 채널 일반화

**목표**: 하하PD(코미디) 외 다른 장르 채널에서도 동일 방법론 검증

### 5-1. 후보 채널

| 채널 | 장르 | 규모 | 비고 |
|------|------|------|------|
| 드나드나 (ENA) | 드라마 | 5.8K | 이미 연결, 분석 3화 완료, 데이터 적지만 장르 다름 |
| 신규 채널 A | 예능/음악 | 10만+ | 추가 OAuth 연결 필요 |
| 신규 채널 B | 브이로그/정보 | 10만+ | 추가 OAuth 연결 필요 |

### 5-2. 할 일

| # | 작업 | 상세 |
|---|------|------|
| E-1 | 드나드나 추가 분석 | `--fast` 모드로 드나드나 전 카탈로그 분석 (드라마 장르 특성 비교) |
| E-2 | 신규 채널 OAuth 연결 | YouTube OAuth 흐름으로 채널 추가 (3~4개 목표) |
| E-3 | 채널 간 비교 | 채널-공통 피처 vs 채널-특화 피처 분리 검증 |
| E-4 | 일반화 보고서 | "코미디·드라마·예능 3개 채널에서 동일한 방법론 검증" NCC 증빙 |

### 5-3. 성공 기준
- 3개 채널에서 각각 매칭 30+ 쌍 확보
- 채널-공통 모델이 모든 채널에서 기준선(Hit@10 ≥ 0.15) 상회

## 6. 트랙 F — 포장 분석 (연구)

**목표**: "같은 구간을 잘라도 조회수 차이가 나는 이유"를 데이터로 밝힌다

### 6-1. 분석 대상
하하PD 고성과 숏폼 26편(ratio≥2) vs 저성과 21편(ratio<0.7)

### 6-2. 분석 축

| 축 | 데이터 | 방법 |
|------|--------|------|
| **제목** | 이미 보유 (catalog CSV) | 이모지·물음표·키워드·길이·어그로 점수 비교 |
| **첫 1~3초 (오프닝 훅)** | 숏폼 영상 다운로드 필요 | 얼굴 클로즈업? 텍스트 오버레이? 강한 리액션? |
| **썸네일** | YouTube API | 색상·텍스트·인물 표정 (비전 필요) |
| **오버레이 텍스트** | 숏폼 영상 필요 | 화면 내 강조 자막·그래픽 패턴 |

### 6-3. 성공 기준
- "이 채널에서 터지는 포장 패턴" 리포트 1건
- 제목: 2~3개 키워드 공통점 발견 / 오프닝: 2~3개 패턴 발견

## 7. 우선순위 및 로드맵

```
주 1 (7/22~7/25)               주 2 (7/28~8/1)               주 3+ (8/4~)
┌─────────────────┐     ┌─────────────────────┐     ┌──────────────────────┐
│ C-① hook 정리     │     │ C-② 가중치 확장       │     │ A-1~4 리텐션 커브     │
│ C-④ few-shot      │────→│ D-1~3 카탈로그 배치  │────→│ B-1~4 폐루프          │
│ A-1 리텐션 현황확인│     │ E-1 드나드나 추가분석│     │ E-2~4 신규채널 연결    │
│ F-1 포장 기본분석  │     │ C-③ 후보생성 적용    │     │ C-⑤ LightGBM          │
└─────────────────┘     └─────────────────────┘     └──────────────────────┘
```

### 당장 오늘 할 수 있는 것 (선행 의존 없음)

| 순위 | 작업 | 예상 시간 | 예상 효과 |
|------|------|-----------|-----------|
| **1** | C-① hook 스키마 enum 강제 (recommend.py) | 30분 | 가중치 신뢰도↑ |
| **2** | C-④ few-shot 프롬프트 (recommend.py) | 1시간 | Hit@10 +5~10%p |
| **3** | A-1 리텐션 커브 현황 확인 (DB 조회) | 10분 | 방향 결정 |
| **4** | F-1 하하PD 고/저성과 제목 비교 (CSV 분석) | 30분 | 포장 인사이트 |
