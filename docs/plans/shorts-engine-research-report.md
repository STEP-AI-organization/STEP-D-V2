## 요약 (Abstract)

"긴 영상에서 자동으로 쇼츠를 뽑는 AI"의 성능을 **재현 가능한 방법론으로 정량화**하고, 100만 구독자 채널(하하PD)의
숏폼 데이터로 학습한 원칙이 **다른 채널(ENA)에도 통하는지** 실증했다.

**핵심 결과 4가지**:
1. **엔진 구조 개선 확인** — 프롬프트·경계·처리속도 결함 3개 발견·수정. 배포됨.
2. **평가 방법론 진화** — IoU(같은 초)의 편향을 발견 → Topic-Hit(내용 매칭)로 교체. IoU는 엔진을 최대 2.6배 과소평가.
3. **채널 학습 컨셉 재정의** — 채널맞춤(오염) → **채널초월 일반화 원칙**으로 전환. 같은 채널 학습·검증의 근본 결함 인지.
4. **크로스채널 일반화 실증** — 하하PD 199 숏폼 학습이 ENA 롱폼에서 **Topic-Hit@10 +20%(평균), 픽 안정성 완전 확보**.

**정직한 한계**: ENA 정답지 7쌍·반복 2회는 여전히 작음. 통계적 확정은 정답지 30쌍+ · 반복 5회+ 후.

## 1. 배경과 문제 정의

### 1-1. 제품 목적
STEP D는 방송사·MCN 대상 클립/쇼츠 스튜디오다. 운영자가 긴 영상을 올리면 AI가 쇼츠 후보 구간을 추천하고,
운영자가 채택하면 트림·인코딩·배포된다. **핵심 질문: "AI가 사람 편집자만큼 잘 뽑는가?"** — 그래야 무인 자동화가 가능하다.

### 1-2. 이 연구가 답하려는 것
- **(a) 잘 뽑나?** 엔진이 실제 편집자 선택을 재현하는가.
- **(b) 학습으로 강화되나?** 인기 채널 데이터로 배운 원칙이 성능을 올리나.
- **(c) 일반화되나?** 한 채널에서 배운 것이 다른 채널에도 통하나.

### 1-3. 왜 지금 이 연구가 필요한가
바우처(NCC) 결과보고 + IR에서 "AI 자동화가 실제로 동작한다"는 것을 **과장 없이 실측 데이터로** 보여야 한다.
직관·개발자 감이 아니라 **재현 가능한 숫자**로.

## 2. 연구 인프라 (재현 가능)

### 2-1. 데이터 파이프라인
```
YouTube 채널 (하하PD·ENA 등) → 오디오/영상 다운로드 (로컬 yt-dlp)
                              ↓
                     Gemini STT (자막) + 시각 분석 (fast 모드는 스킵)
                              ↓
                    recommend (2단계: 후보 추출 → 합성)
                              ↓
                         쇼츠 후보 (top-N)
```

### 2-2. 정답지 구축 (Ground Truth)
- **원칙**: 채널이 **실제로 발행한 숏폼** = 사람 편집자 선택 = 정답. 가장 신뢰 높은 정답 원천.
- **역추적**: `core/align.py` (오디오 상호상관 FFT). 숏폼 오디오가 롱폼 어느 구간에서 나왔는지 계산.
- **매칭 신뢰도 실측**: 진짜 매칭 score 0.886~0.998 · 무관 0.403~0.601 (간격 넓어 오탐·미탐 여유)
- **성과 정규화**: 절대 조회수 금지. ±90일 동시기 채널 중앙값 대비 배수(ratio) → tier (high ≥2, mid ≥0.7, low <0.7)

### 2-3. 평가 지표 (진화)
| 지표 | 판정 방법 | 언제 씀 |
|---|---|---|
| **Hit@N (IoU≥0.5)** | 엔진 top-N과 정답 구간이 50% 이상 겹치나 | Exp 1·2 (초기) |
| **Topic-Hit@N** | Gemini가 "엔진 픽 = 정답과 같은 순간/주제인가" 내용 판정 | Exp 3·4 (개선) |

**왜 지표를 바꿨나** (사용자 지적): IoU는 8초만 어긋나도 오답 → 엔진이 같은 순간을 잡고도 위치가 조금 다르면
탈락. 제품이 진짜 묻는 건 "같은 것을 숏폼감으로 봤나"이지 "같은 초에 잘랐나"가 아님.

### 2-4. 재현성 도구
- `core/eval_repeat.py` — IoU 기반 N회 반복 채점 (엔진 변동 상쇄, 평균±σ)
- `core/eval_topic.py` — Topic 기반 내용 매칭
- `core/eval_topic_ab.py` — off/base/base+예시 A/B
- `core/scenes.py:scenes_from_transcript` — 자막 코스 유닛 (마이크로컷 회피)

## 4. 엔진 구조 개선 (실측 발견·수정)

배포된 코드 개선. 학습·프로파일과 무관하게 모든 채널에 유효.

### 4-1. 너무 짧게 잘라내던 문제
- **발견**: 첫 홀드아웃 채점에서 Hit@5 = 0.00. 원인은 "못 찾음"이 아니라 **4~8초로 잘라 IoU 붕괴**.
- **원인**: 프롬프트 모순 ("펀치라인만/짧게"가 "30~60초" 지시를 덮음)
- **수정**: 프롬프트 재작성 + 30초 하한(짧으면 장면 경계로 확장). 실측: 8초→45초. `core/recommend.py:_extend_to_min`
- **효과 (측정)**: 클립 CSV 173편의 **97%가 30~60초** — 길이 개선 실전 반영.

### 4-2. 대사 중간에서 끊기던 문제
- **발견**: 클립이 "갑자기 대사 시작"·"말 뚝 끊김"이 남 (자막 검토 시 다수).
- **원인**: 경계를 장면(시각 컷)에만 스냅했음. 시각 컷은 말하는 도중일 수 있음.
- **수정**: **발화(문장) 타임스탬프로 스냅** — 시작이 대사 도중이면 문장 처음으로, 끝이 도중이면 문장 끝까지 (2.5초 밖이면 손 안 댐). AI·휴리스틱 두 경로 모두 적용. `core/recommend.py:_snap_to_speech`, 단위테스트 6/6 통과.

### 4-3. 처리 속도 (긴 영상 분석 실패)
- **발견**: 원정대5(61분) 풀 파이프라인 분석 시 30분 무출력으로 워커 stall 워치독이 강제 종료.
- **실측**: 시각 장면감지+프레임 분석이 긴 영상 시간의 **최대 74%**(13분 영상이 63분).
- **수정**: **fast 모드**(`--fast`) — 자막 세그먼트로 후보 구간 구성, 시각 분석 스킵. **61분 영상 → 96초** (원정대5), 35분 → 68초 (원정대4). `core/analyze.py`, `core/scenes.py:scenes_from_transcript`
- **부수 수정**:
  - 장면감지 하트비트 (긴 영상 stall 오판 방지)
  - 받아쓰기 degeneration 정제 ("모 모 모..." 반복 제거, 실측 DB 4건 최대 4032자 오염 교정)
  - audio-only 다운로드 (fast는 오디오만 → 5-10배 빠름)

## 5. 실험 (Exp 1-4)

### Exp 1 — 엔진 개선 검증 (단일 홀드아웃, 2026-07-21)
- **방법**: 하하PD 경주 PC방(13분) 홀드아웃, 정답 3, IoU Hit@5
- **결과**: Hit@5 0.00 → **0.67** (개선 전/후)
- **의미**: 엔진 구조 개선(§4-1)의 효과 확인
- **⚠️ 함정 (F2)**: 정답 3개는 해상도 4단계(0/0.33/0.67/1)뿐 → 표본 확대 시 정정 필요 확인

### Exp 2 — 다중 홀드아웃 반복 A/B (IoU, 2026-07-21)
- **방법**: 홀드아웃 3편 (정답 26) × 학습 on/off × 5회 반복
- **결과**:

  | 지표 | 학습 끔 | 학습 켬 | Δ | 판정 (차이 vs σ) |
  |---|---|---|---|---|
  | Hit@5 | 0.100 | 0.138 | +38% | 유의 |
  | Hit@10 | 0.131 | 0.200 | +53% | **미확정** (σ 안) |
  | Hit@20 | 0.154 | 0.215 | +40% | 유의 |

- **핵심 발견**: recommend가 temp0인데도 **실행마다 결과 요동** (같은 조건 5회: 0.00~0.67).
  → 단발 A/B로 학습 효과 판정 불가. **반복 평균 필수화**.
- **정직한 한계**: 절대 Hit@10 = 0.20 → 무인 발행 수준 아님. IoU 지표의 한계는 아직 못 봄.

### Exp 3 — Topic 지표 도입 (2026-07-22, 뒤집힘)
- **방법 변경**: IoU → **Topic-Hit@N** (Gemini 내용 매칭). "같은 초"가 아니라 "같은 순간".
- **추가**: ④ few-shot 예시 프롬프트 (규칙 대신 원본 예시 삽입)
- **결과 (2회 반복, 마이크로컷 코스화 후)**:

  | 조건 | Topic-Hit@10 |
  |---|---|
  | off | 0.519 |
  | base (규칙) | 0.346 |
  | base+예시 (④) | 0.346 |

- **큰 발견 (사용자 지적)**:
  - IoU가 엔진을 크게 과소평가 (off Topic 0.52 vs IoU 0.20 = **2.6배 격차**)
  - 학습이 오히려 성능 -33% 악화? Exp 2와 정반대
- **감사 (F3)**: 재실행 시 3조건 다 0.667 (해상도 부족). **결론 신뢰불가로 재분류**.
- **근본 결함 인지**:
  - (a) 학습·검증 **같은 채널**(하하) → 오염
  - (b) 표본 26정답×2회 → σ가 지표 차이 삼킴
  - (c) LLM 판정관 detail 미저장 → 감사 불가
- **의미**: Exp 3은 **"학습 방식 결함 발견"으로만 유효**. → Exp 4에서 컨셉 재정의.

### Exp 4 — 채널초월 일반화 + 크로스채널 실증 (2026-07-22, 진짜 진전) ⭐
- **컨셉 전환 (사용자 방향)**: "채널맞춤 학습" → **"일반적으로 잘 터지는 쇼츠 요소 학습"**.
  롱폼↔숏폼 매칭(오디오정렬 병목) 대신 **하하 발행 숏폼 자체를 학습 소스로**.

#### Phase 1 — 하하 숏폼 199 학습 데이터 구축
- Phase 1a: 오디오 다운로드 **199/228** (fail 29=삭제/비공개, 로컬 봇차단 0)
- Phase 1b: STT **199/199 fail 0** (63분, Gemini, 순차 tmp 충돌 회피)
- Phase 1c: 시각 요소는 후단계 (오디오만 확보)

#### Phase 2 — 요소 대조 학습
**dur≤180초 재필터 후 표본** (F7 발견): high 69 · mid 69 · low 69

| 요소 | high | low | lift | 신호 |
|---|---|---|---|---|
| **n_hooks (제목 어그로)** | 0.13 | 0.07 | **1.86** | 최강 |
| **emoji** | 0.13 | 0.09 | **1.44** | 강 |
| has_quote | 0.04 | 0.03 | 1.33 | 약 |
| title_len | 28.6자 | 24.9자 | 1.15 | 약 |
| **dur_sec** | 44초 | 54초 | **0.82** | 강 (짧을수록) |

| dialog_dens | 7.8 | 8.05 | 0.97 | 무신호 |

**일반화 원칙**: "**짧고(45초), 대사 적고, 제목이 자극적**"인 하하 숏폼이 잘 터진다.

#### Phase 3 — 크로스채널 검증 (진짜 답)
- 하하학습 프로파일(위 원칙만 담음, hookWeights 편중 제거 = Exp 3 반성)을 **ENA 롱폼에 적용**.
- ENA 정답지: 30 숏폼 × 5 롱폼 오디오매칭 → **7쌍 확보** (4 롱폼 커버, score 0.80~0.99)
- 각 조건 2회 독립 실행:

  | 조건 | 1회차 | 2회차 | **평균** | 범위 |
  |---|---|---|---|---|
  | off | 0.571 | 0.857 | **0.714** | 0.571~0.857 (**요동**) |
  | **하하학습 on** | 0.857 | 0.857 | **0.857** | 0.857 (**안정**) |

- **결정 사례**: ENA 롱폼 QNtoQ4zI8mc — off @10=0.0, 하하학습 @10=1.0. 순수 엔진이 놓친 편집자 선택을 학습이 찾아냄.

#### Exp 4 결론 (두 발견 분리)
1. **평균 개선 (약함)** ✅: 학습 켬 평균 @10=0.857 vs off 0.714 → **+20%**. 표본 7쌍·반복 2회라 통계 확정은 아님.
2. **안정성 격차 (더 중요)** ⭐: 학습 켬은 **두 번 다 0.857 (동일)**. off는 0.571~0.857 요동.
   **일관성이 학습의 진짜 가치** — 실전에서 매번 다른 결과 나오면 못 씀.
3. **크로스채널 원칙 실증**: 하하 100만 채널의 숏폼 요소 학습이 다른 채널(ENA)에도 통함을 확인.

## 8. 결론

### 8-1. 이 연구가 실증한 것
1. **엔진의 구조적 결함 3가지를 발견·수정**: 길이·경계·처리속도. 프로덕션 배포됨.
2. **재현 가능한 측정 방법론 확립**: 발행 숏폼 정답지 + Topic 기반 채점 + 다회 반복 + 크로스채널 검증.
3. **채널 학습의 본질을 발견**: "채널맞춤"은 오염되고 다른 채널에 편향. **"채널초월 일반화 원칙"(짧게/대사적게/제목자극)**이 실효 있음.
4. **크로스채널 일반화 실증**: 하하PD 학습 → ENA에서 **+20% 평균, 안정성 격차** 확인. 완전 무인은 아직이나 방향 확정.

### 8-2. 이 연구가 실증하지 못한 것 (정직한 한계)
- ENA 정답지 7쌍은 **통계 확정에 부족**. 최소 30쌍+ 필요.
- 각 조건 2회 반복 = σ 정확 추정 못함. 5회+ 필요.
- ENA도 예능 도메인이라 하하와 겹침 — **다른 도메인**(정보·리뷰·다큐)까지는 미검증.
- 시각 요소(오프닝 프레임·오버레이·자막 스타일)는 학습에서 제외. 하하가 100만인 이유가 여기 있을 수도.
- 성과 tier = 조회수 = **포장(제목·썸네일) 영향 큼**. "잘 자름"과 "잘 포장"이 아직 분리 안 됨.

### 8-3. 방법론적 진전 (이 연구의 부산물)
- **엔진 실행 변동 발견** (temp0 LLM도 요동) → 반복 평균의 필수성 확립
- **IoU 지표의 편향 발견** → Topic 기반 채점 도입
- **fast 모드 개발** → 실험 반복이 현실화 (61분→96초)
- **로컬 실험 환경** → 봇차단 회피, 실험 신뢰도 확보
- **실패·폐기 실험을 정직하게 기록** (F1~F7) → 같은 실수 반복 방지
