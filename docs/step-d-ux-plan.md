## 0. 한눈에 보기 (Executive Summary)

## 1. 배경 — STEPD는 무엇인가

**제품 한 줄:** KT ENA 방송 콘텐츠를 `VOD 소스 → 완성본(클립/쇼츠/미디어) → 다채널 배포(네이버 SMR · YouTube · Meta Reels)`로 가공·배포하는 **관리자용 스튜디오 툴**. 한국어, 라이트 톤 기본. 역할 `superadmin / admin / user`.

**핵심 파이프라인 (좌→우):**
```
[소스 관리]                                   [완성본]              [배포]
프로그램 → 회차 → MXF소스 → (자동) 병합→분할→Gemini업로드→3-Pass분석→추천
                                                          │
                                          ── (사람 개입 경계) ──
                                                          ▼
                              편집(구간+오버레이) → 인코딩(export) → 등록(register)
                                                          ▼
                                    SMR / YouTube / Meta / 배포 스케줄
```
- `merge → split → gemini-upload → analyze → recommend`까지 **잡 큐(pg-boss) 기반 완전 자동 체이닝**. 이후 `편집 → export → register → 배포`는 **대부분 수동 트리거**. 이 "수동 경계"가 채택률 저하의 핵심 원인 중 하나. (`PIPELINE_OVERVIEW.md §4-2`)

(관계는 대부분 문자열 값 매핑, DB FK 거의 없음.)

## 2. 현재 STEPD UX 페인포인트 (실무자·운영자 관점)

> STEPD 개발팀 자체 진단(`ROADMAP_2026H2.md`, `WORKLOG_2026-06-29/07-09.md`)을 UX 관점으로 재구성. 각 항목에 근거 문서를 붙였다. v2 설계는 이 목록을 정면으로 해소하는 것을 목표로 한다.

### A. 워크플로우가 파편화됨 — "엔티티 CRUD"라 업무 흐름과 어긋남
- **A1. 추천→클립 원클릭 채택이 없음.** 추천 1건을 실제 클립으로 만들려면 `편집기 진입 → 구간조정 → 오버레이 → 인코딩 큐(export) → 등록(register) → 배포`를 전부 수동으로 밟아야 함. **쇼츠 채택률 ≈0의 1순위 원인.** (`WORKLOG_2026-06-29 §2`, `PIPELINE_OVERVIEW §3-3`)
- **A2. 화면이 엔티티별로 흩어져 있음.** 사이드바가 프로그램/회차/소스/인코딩/미디어/클립/SMR/YT/Meta/스케줄… 각각 목록·상세·폼. **"이 회차가 지금 파이프라인 어디에 있나"를 한 화면에서 볼 수 없음.** 콘텐츠 1건을 밀려면 화면을 계속 갈아탐.

### B. 의사결정 지원 부재 — 운영자를 혼자 둠
- **B1. 후보 과다로 결정 피로.** 소스당 쇼츠 15~20개가 우선순위 없이 쏟아짐. (`WORKLOG_2026-06-29 §2`)
- **B2. 거절/사유 기록 없음 → 피드백 루프 부재.** "이건 별로"라고 치우고 학습시킬 수단이 없음.
- **B3. recommendations / clips / distributions 3테이블 단절.** 추천→클립→배포→성과 계보를 못 따라감 → 추적·측정 불가.
- **B4. 성과 지표 자체가 없음.** `distributions`에 조회수/수익 컬럼 없음, 채택률 대시보드 없음. 운영자가 눈 가리고 일함. (`PIPELINE_OVERVIEW §3-4`)

### C. 가시성·신뢰 문제 — 실제로 무슨 일이 일어나는지 안 보임
- **C1. 파이프라인 상태가 불투명.** 긴 비동기 잡이 백그라운드로 돎. "어디까지 갔나 / 뭐가 막혔나 / 뭐가 내 액션을 기다리나"를 보여주는 라이브 뷰가 약함. G-RAID 장애로 유튜브 14건이 죽었을 때도 **알림 없이** 운영자가 알아서 재클릭. (`WORKLOG_2026-06-29 §1`)
- **C2. UI 문구와 실제 동작이 반대(신뢰 붕괴).** SMR "즉시발행: 비워두면 즉시 노출" → 실제론 예약일 빈값 = **미게시**. (`WORKLOG_2026-07-09 §1`)
- **C3. 데드엔드 UI.** 클립 페이지가 소프트삭제된 SMR을 "배포중"으로 표시 → 재배포 불가 무한루프. (`WORKLOG_2026-07-09 §3`)
- **C4. 필터/상태가 유지 안 됨.** 필터가 `useState`만 써서 새로고침·재진입 시 초기화. 컨텍스트를 계속 잃음. (`ROADMAP §1-6`)
- **C5. '표시만 되고 액션 없는' 죽은 UI.** 추천 썸네일 후보가 보이기만 하고 "이걸로 쓰기" 버튼 없음 / 유튜브 카테고리 항상 '24' 고정. (`ROADMAP §3-2, 빠른승리`)

### D. 편집기 UX가 약함 — STEPD 스스로도 리뉴얼 대상(로드맵 E)
- **D1. 에디터 UI/UX 리뉴얼이 "향후 과제 E"로 밀려 있음** — 로드맵에 *"CapCut 벤치마킹"* 명시. (`ROADMAP 기타/향후`)
- **D2. 말자막(STT) 자체가 없음.** 분석은 Gemini 영상분석만. (`ROADMAP §2-2`)
- **D3. 템플릿/복제/폰트 확장 없음.** 폰트 1종(HGGGothicssi), 오버레이 템플릿 저장·적용 없음, 클립 복제 없음. (`ROADMAP §2-1, §2-3`)
- **D4. 싱크 오차 미세조정 UI 없음.** 반올림 누적으로 추천 구간 타임스탬프가 어긋나는데 ±초 조정 수단 없음. (`ROADMAP §1-4`)

### E. 배포·스케줄 UX 파편화
- **E1. 채널마다 별도 화면·별도 모달**(SMR/YT/Meta), quirk도 제각각. "이 클립을 이 채널들에 한 번에" 같은 통합 배포 액션이 없음.
- **E2. 스케줄러가 단순 CRUD.** recurrence/bulk/템플릿/플랫폼 분리 없음. (`ROADMAP §3-7`)

## 3. StepD에서 가져올 강점

> 출처: `INTEGRATION_LONGPONG_ANALYSIS.md`(STEPD 내 통합분석), StepD `apps/web/app/components/ShortcutEditor.tsx`, `apps/docs/{product-vision,feature-status}.md`, `apps/api/app/services/*`.

StepD는 "긴 영상 → 쇼츠 → 유튜브"라는 **동일 목표를 다른 스택(Python/FastAPI)으로 구현**한 제품. STEPD와 기능이 70%+ 겹치되, **쇼츠 생성 품질과 에디터 UX**에서 앞선다. v2는 StepD를 통째로 이식하는 게 아니라 **UX 패턴과 개념**을 흡수한다.

### 3.1 에디터 UX (v2 에디터의 청사진)
`ShortcutEditor`에서 검증된, STEPD 에디터에 없는 강점들:
1. **장르별 템플릿 프리셋(5종)** — `상단 제목+채널` / `풀스크린 세로` / `캡션 카드`(토크·예능) / `뉴스 바`(정보·이슈) / `댓글 유도 훅`. 각 프리셋이 종횡비·배경·강조색·제목/자막 위치·폰트·채널표시를 한 번에 세팅 + 한 줄 힌트("가로 원본을 쇼츠로 가장 무난하게"). → **원클릭 레이아웃**. (STEPD 로드맵 §2-1이 원하던 것)
2. **WYSIWYG 정확 프리뷰** — 프리뷰 캔버스를 9:16·720px에 고정하고 **프리뷰 px → 렌더 px 스케일**, 텍스트 줄바꿈을 ffmpeg 번인과 동일한 폰트 메트릭으로 계산(`wrapOverlayText`). **"보는 대로 = 나오는 대로."** → STEPD의 "프리뷰≠출력" 불신 해소.
3. **탭형 편집 패널** — `텍스트 / 채널 / 레이아웃 / 자막 / 요소` 5탭. 밀도 높은 단일 패널 대신 작업 단위로 분리.
4. **라인별 제목 스타일링** — LLM 생성 쇼츠 제목을 시드로, 각 줄마다 크기·색·자간·폰트 개별 조정.
5. **종횡비 즉시 전환**(16:9/4:3/1:1/4:5/3:4/9:16) + 레이아웃 자동 재배치.
6. **훅 도구** — `첫 3초 훅` / `배속(리듬 보정)` / `무음 제거`. 참여율 지향 편집 보조.
7. **요소 오버레이** — CTA·스티커·화살표·말풍선·이미지 등 바이럴 요소 즉시 삽입.
8. **채널 브랜딩 오버레이** — 채널명 + 원본 제목 + 아이콘.
9. **에디터 내 배포 메타데이터 팝오버** — 지금 만드는 쇼츠의 유튜브 업로드 메타(제목·해시태그·설명·공개설정·추천 발행시각·예측 성과)를 편집 중에 바로 확인 → 배포가 맥락적.
10. **가라오케 자막 미리보기** — 실제 `<video>` + `requestAnimationFrame` 루프로 재생 중 단어별 하이라이트(훅 단어 강조). 서버 렌더 왕복 없이 "틱톡 자막" 느낌.
11. **YouTube Shorts UI 세이프에어리어 오버레이** — 좋아요/댓글/구독 크롬을 목업으로 얹어 안전 영역을 보여줌(저비용·고체감).

**v2가 그대로 채택할 엔지니어링 계약(패턴):**
- **선언적 `EditorState` 통짜 직렬화** — 에디터는 명령형 편집 커맨드가 아니라 하나의 상태 객체(레이아웃/오버레이/트림/트랜스폼)를 통째로 저장하고, **렌더러가 그 상태로부터 최종 출력을 재유도**. (재현성·자동저장·재렌더 캐시에 유리)
- **퍼센트 좌표 + 고정 종횡비 프리뷰 캔버스, 렌더러가 같은 px 기준을 공유** — 오버레이 좌표를 %로 저장(종횡비 바꿔도 생존), 프리뷰(예: 9:16·720px) px ↔ 렌더 px를 동일 기준으로 스케일. **이것이 WYSIWYG 일치의 핵심.** (STEPD도 이미 서버 오버레이 렌더+해시 캐시가 있어 결합 용이)
- **문장 종결어미 컷 스냅** — 정규식 기반(다/요/죠/네/야/지/까… + 종결부호)으로 대사 중간 안 잘림. 가장 저비용·고가치 휴리스틱, 이식 쉬움.
- **이미 자막 있으면 건너뛰기** — 자막 스트림 프로브 + 번인 자막 검출(신뢰도 임계)로 이중 자막 방지.
- **스코어 블렌드는 코드가 정본** — StepD는 문서상 "82/18"이나 실제 코드는 `gemini*0.70 + local*0.30`. v2가 스코어링을 참고하면 **코드 값(70/30)** 을 따른다.

### 3.2 쇼츠 생성 품질 (파이프라인 흡수 후보)
`feature-status.md` 기준 StepD가 완성한 것 중 STEPD에 없는/약한 것:
- **STT 기반 후보 검출**(Whisper 한국어) + 훅 키워드 기반 하이라이트 — 말 많은 예능/토크형에 유리.
- **한국어 종결어미 컷 스냅**(다/요/죠/네 + 마침표) — 대사 중간 안 잘리는 자연스러운 컷.
- **9:16 세로 자막 번인**(G마켓 산스) + 블러 배경.
- **번인 자막 검출** — 원본에 자막 있으면 이중 자막 방지.
- **무음 구간 탐지 → 편집 제안 탭**(타임코드·절약시간 표시).
- **바이럴 제목 5종 + 썸네일 텍스트 옵션** 생성.
- **yt-dlp 유튜브 URL 임포트** — 외부 소스 인입.

### 3.3 제품 개념 (IA·리뷰 흐름에 반영)
`product-vision.md`의 핵심 사고 = **"영상을 시간축 데이터로 구조화"** → 검수·편집·PPL·승인·성과에 재사용. v2가 흡수할 개념:
- **타임코드 이슈 카드** — 쇼츠 후보·무음·PPL 노출·개인정보·음량 피크를 **하나의 타임라인에** 얹고 검수/편집/리포트에서 재사용.
- **승인/반려/수정요청 리뷰 흐름 + 타임코드 댓글** — STEPD 페인포인트 B2(거절·사유)·B3(계보)·C1(가시성)을 정면 해소.
- **성과 → 피드백 루프** — 잘된 훅/자막/출연자/PPL 패턴 누적 → 다음 추천에 반영.

> **주의(안전):** StepD 코드는 참고만 하고 포팅 시 v2에서 재구현한다. StepD 저장소는 절대 수정하지 않는다. STT는 OpenAI 의존이라 이식 비용이 크므로 **초기엔 개념·UI만 반영하고 실제 STT 연동은 후반/사이드카로 결정**(통합분석 C안).

## 4. 설계 목표 & 원칙

**최상위 목표:** *"운영자가 지금 무엇을 해야 하는지 한눈에 알고, 최소 클릭으로 회차를 소스에서 배포까지 민다."*

**설계 원칙 (v2 전 화면에 적용):**
1. **작업 중심 > 엔티티 중심.** 첫 질문은 언제나 "무엇이 내 액션을 기다리나?". (해소: A2, C1)
2. **원클릭 채택.** 추천 → (채택) → 자동으로 export+register 체인 → 편집은 선택. (해소: A1, B1)
3. **의사결정 지원.** 우선순위(appeal 랭킹)·정예 추천(top N)·채택/반려+사유·계보 추적을 기본 제공. (해소: B1~B4)
4. **상태 정직성(Truth-first).** 프리뷰 = 출력, UI 문구 = 실제 동작. 모호한 "즉시발행" 같은 문구 금지, 실제 결과를 미리 보여줌. (해소: C2)
5. **데드엔드 제로.** 화면에 보이는 모든 것은 액션 가능하거나 다음 단계로 이어짐. (해소: C3, C5)
6. **상태는 색·배지로 즉시 식별** + **필터/뷰는 URL 동기화·저장.** (해소: C4)
7. **밀도는 유지, 위계는 재정립.** 소비자 앱이 아니라 프로 도구 — 표·필터·배지·키보드가 핵심. 여백보다 스캔 가능성.
8. **키보드 우선.** 전역 커맨드 팔레트(⌘K), 에디터 단축키(J/K/L·I/O·스페이스 등).
9. **라이브·낙관적.** 잡 진행률(SSE) 상시 노출, 액션은 낙관적 반영 후 확정.
10. **점진적 노출.** 합리적 기본값 먼저, 고급 옵션은 접힘.
11. **한국어 우선, 라이트/다크 자동, 접근성 준수**(대비·포커스링·ARIA).

## 5. 정보구조(IA) 재설계

### 5.1 현재 vs v2 (핵심 전환)

| | 현재 STEPD | STEP-D |
|---|---|---|
| 조직 원리 | 엔티티 CRUD(프로그램/회차/소스/클립…) | **작업 단계 + 대상 객체**(할 일 → 회차 진행 → 추천 채택 → 편집 → 배포 → 성과) |
| 진입점 | 통계 대시보드 | **작업 큐("무엇이 내 액션을 기다리나")** |
| 회차 단위 뷰 | 없음(엔티티별로 흩어짐) | **회차 상세 = 파이프라인 진행 스트립 단일 화면** |
| 추천 | 소스 상세에 15~20개 나열 | **추천 리뷰 보드**(랭킹·정예·채택/반려+사유·원클릭) |
| 배포 | 채널별 별도 페이지·모달 | **통합 배포(멀티채널 한 번)** + 스케줄러 개편 |
| 성과 | 없음 | **계보·채택률·성과 대시보드** |

### 5.2 v2 최상위 내비게이션 (7 영역)

```
■ 홈 / 오늘 할 일 (Inbox)        ← "무엇이 내 액션을 기다리나"
■ 콘텐츠 (Programs → Episodes)   ← 회차별 파이프라인 진행 단일 화면
■ 추천 & 채택 (Recommendations)  ← 리뷰 보드 + 원클릭 채택
■ 에디터 (Editor)                ← 풀스크린, StepD급 (별도 셸)
■ 배포 (Distribution)            ← 통합 멀티채널 + 스케줄
■ 성과 & 추적 (Analytics)        ← 계보·채택률·채널 성과
■ 시스템 (admin)                 ← 채널 연동·계정·파일·파이프라인 헬스
```

- **소스 관리(MXF/인코딩)**는 별도 최상위에서 **회차(콘텐츠) 하위로 흡수** — 운영자에게 "소스"는 회차의 한 단계일 뿐, 별도 목적지가 아니다.
- **완성본(미디어/클립)**은 **콘텐츠·추천·배포 안에서** 자연스럽게 등장(전용 "클립 목록"은 필터 뷰로 유지하되 부차적).
- 역할별 노출: `user`=홈·콘텐츠·추천·에디터·배포, `admin`=+성과·시스템 일부, `superadmin`=+계정·전체 시스템.

## 6. 내비게이션 셸 재설계

- **좌측 사이드바(재편):** 위 7영역을 작업 순서대로. 각 항목에 **미해결 카운트 배지**(예: 추천 채택 대기 12, 배포 실패 3). 접기 가능(아이콘 레일).
- **상단 앱바:** 좌측 = 현재 위치 브레드크럼(프로그램 › 회차 › 단계), 우측 = **전역 검색/커맨드 팔레트(⌘K)**, **잡/알림 센터**(진행 중·막힘·실패를 벨 배지로), 테마·계정.
- **잡/알림 센터(신규, 전역):** 실행 중 잡·진행률·실패·"당신의 액션 필요"를 한곳에. C1(가시성)·C3(데드엔드) 해소의 핵심. 실패 항목은 바로 "재시도"/"재배포" 가능.
- **커맨드 팔레트(⌘K):** 회차·클립·프로그램 이동, "이 회차 추천 채택", "배포", "에디터 열기" 등 액션 직행.
- **필터/뷰 지속:** 모든 목록은 `searchParams` 동기화 + "저장된 뷰"(예: "채택 대기 · 예능 · 이번주"). (C4 해소)
- **인앱 도움말 드로어 유지·개선:** STEPD `studio-help` 패턴을 화면별 컨텍스트 가이드로 계승.

## 7. 핵심 화면 설계

### 7.1 홈 / 오늘 할 일 (Inbox) — `/`
"통계 대시보드"가 아니라 **액션 큐**. 섹션:
- **내 액션 필요:** ⬥ 채택 대기 추천(정예 top) · ⬥ 편집 대기 · ⬥ 인코딩 완료→등록 대기 · ⬥ 배포 대기(ready) · ⬥ **배포 실패/막힌 잡**(원클릭 재시도).
- **진행 중:** 자동 파이프라인 도는 회차(진행률 스트립). 
- **최근 성과 요약**(admin): 채택률·최근 배포 성과(연결되면).
- 각 카드는 **바로 그 자리에서** 다음 액션 수행(채택/등록/재배포/에디터 열기) — 목록으로 튕기지 않음.

### 7.2 콘텐츠: 프로그램 → 회차 → **회차 상세(파이프라인 허브)**
- **프로그램 목록/상세:** 메타(출연/장르/시청연령/장르팩 aiConfig/인트로·아웃트로) + 소속 회차. 필터 URL 동기화.
- **회차 상세 = v2의 심장.** 한 화면에서 회차의 전 여정을 보여준다:
  ```
  [회차 헤더: 프로그램·회차번호·방송일·시청연령]
  [파이프라인 진행 스트립]
   소스 ─● 병합 ─● 분할 ─● 분석 ─◐ 추천 ─○ 편집 ─○ 인코딩 ─○ 배포
        (각 단계: 상태 배지·진행률·소요·오류·"이 단계 실행/재시도" 액션)
  [탭] 소스 파일 | 분석 결과 | 추천(→채택) | 클립(완성본) | 배포 현황
  ```
  - "이 회차 지금 어디까지?"를 즉답(A2 해소). 막힌 단계는 붉은 배지 + 액션.
  - 소스 인제스트(MXF 업로드/로컬 임포트/**yt-dlp URL 임포트(신규)**)도 여기서.

### 7.3 추천 & 채택 (Recommendation Review Board)
STEPD 페인포인트 B의 정면 해소. StepD "이슈 카드" 개념 차용.
- **정예 우선:** 15~20개 그대로 쏟지 않고 **appeal≥4 top N**을 상단에, 나머지는 접힘("더 보기"). (B1)
- **추천 카드:** 썸네일(후보 프레임 선택 가능·C5 해소)·제목·appeal 배지·구간(시:분:초·길이)·유형(쇼츠/클립)·인물/브랜드 태그·editNote·미리보기.
- **원클릭 "채택":** export+register 자동 체인 → 클립 생성. "채택 후 편집"은 선택. (A1) 진행은 잡 센터에 표시.
- **반려 + 사유:** 반려 사유(품질/중복/부적합…) 기록 → 피드백 루프·측정. (B2)
- **계보 링크:** 채택된 추천 → 생성 클립 → 배포 → 성과로 이어지는 lineage 표시. (B3)
- 타임라인 오버레이(쇼츠 후보·무음·PPL 구간)로 확장 여지(StepD 개념).

### 7.4 에디터 (Editor v2) — `/editor/[projectId]` (풀스크린 셸)
StepD `ShortcutEditor` + STEPD 에디터(트랙/세그먼트/서버 오버레이 렌더 캐시)의 **장점 결합**. 다크 톤.
- **레이아웃:** 좌 = AI 패널(분석/클립/쇼츠 후보, 클릭 시 참조 트랙) · 중앙 = **WYSIWYG 정확 프리뷰**(고정 해상도 캔버스, 프리뷰px→렌더px 스케일) · 우 = **탭형 편집 패널**(텍스트/채널/레이아웃/자막/요소) · 하단 = **타임라인**(구간·IN/OUT·세그먼트·줌·참조 구간·PPL/무음 오버레이).
- **템플릿 프리셋(신규):** 장르별 원클릭 레이아웃 + 저장/적용(로드맵 §2-1). (D3)
- **자동 자막(신규):** STT 트랜스크립트 → 읽기 좋은 줄 분할 → 자막 오버레이 자동, 텍스트·스타일 커스텀. (D2)
- **종횡비 전환** + 오버레이 좌표 리스케일(STEPD `9:16-crop-*`/`letterbox` 프리셋 계승).
- **훅 도구:** 첫3초 훅·배속·무음 제거 제안. **±초 미세조정**으로 싱크 교정(D4).
- **요소:** CTA·스티커·화살표·말풍선·이미지. **폰트 확장**(G마켓 산스 등, HGGGothicssi 단일 탈피). **클립 복제**(원본 유지 변형). (D3)
- **자동저장**(디바운스) + 서버 오버레이 렌더 캐시(프리뷰=출력 동일 에셋).
- **에디터 내 배포 메타데이터 미리보기**(StepD) → 배포로 매끄럽게 연결.
- **키보드:** 스페이스(재생)·I/O(IN/OUT)·J/K/L·←/→(프레임/넛지)·Del(세그먼트 삭제)·⌘S.

### 7.5 배포 (통합 멀티채널) — `/distribution`
- **통합 배포 액션:** 클립(들) 선택 → **한 다이얼로그에서 SMR·YouTube·Meta 동시 설정**(채널별 필드는 탭/섹션). 채널별 quirk를 UI가 흡수. (E1)
  - **정직한 예약:** "즉시발행"의 실제 동작(예약일=현재시각 자동)을 미리 보여주고 라벨을 실제와 일치시킴. (C2)
  - 카테고리·시청연령 등 자동 프리필 + 편집 가능(카테고리 '24' 고정 해소·C5).
- **배포 현황 보드:** 채널별 상태(대기/예약/게시/실패)를 클립×채널 매트릭스로. 소프트삭제/재배포 상태를 `isUse` 기준으로 정확히 판정(C3 해소).
- **스케줄러 개편:** 반복(recurrence)·일괄(bulk)·템플릿·플랫폼 분리. 회차×채널 일괄 예약. (E2)

### 7.6 성과 & 추적 (Analytics / Lineage) — `/analytics`
- **채택률 대시보드:** 추천 생성→채택→배포 전환율(B4). 반려 사유 분포.
- **계보 뷰:** recommendation → clip → distribution → (성과) 한 줄로. (B3)
- **채널 성과:** YouTube 조회수/댓글/완주율 등(스코프 확장 시). StepD의 댓글 AI 요약 개념 참고.
- *주의:* 현재 STEPD `distributions`엔 성과 컬럼이 없음 → v2는 UI를 먼저 만들고 데이터는 백엔드 확장/외부 API 연결 시 채운다(마일스톤 M5).

## 8. 핵심 플로우

**플로우 1 — 인제스트→추천(대부분 자동):**
회차 생성 → 소스 업로드(MXF/로컬/URL) → (자동) 병합→분할→분석→추천 → **홈 Inbox와 회차 스트립에 진행률·완료 알림**. 운영자는 지켜보기만; 막히면 잡 센터가 알림 + 재시도.

**플로우 2 — 추천→채택→편집→인코딩(원클릭 경로):**
추천 리뷰 보드 → 정예 카드 확인 → **[채택]**(export+register 자동) → (선택) 에디터에서 다듬기 → 인코딩 완료 → 클립 ready. 반려는 사유와 함께 한 번에.

**플로우 3 — 배포(통합):**
ready 클립 선택 → **[배포]** 통합 다이얼로그(SMR+YT+Meta 동시, 예약 정직 표시) → 배포 현황 보드에서 채널별 상태 추적 → 실패는 그 자리에서 재배포.

**플로우 4 — 추적/성과:**
채택→배포→성과 계보 확인, 채택률·반려사유로 추천 품질 개선 루프.

## 9. 디자인 시스템 & 비주얼 방향

- **기반:** STEPD와 동일 — **shadcn/base-ui**(`@base-ui/react`), style `base-nova`, baseColor `neutral`, CSS 변수(HSL) 토큰, Tailwind v4 `@theme inline`, lucide 아이콘. → 엔지니어가 1:1로 실서비스에 반영 가능, 학습비용 0.

- **공유 컴포넌트 레이어(신규 표준화):** STEPD는 대부분 인라인 Tailwind로 UI를 짬(공유 프리미티브 3개뿐: button/select/alert-dialog). v2는 초기에 **Table·Badge/StatusChip·Dialog·StatCard·Field·PipelineStrip·RecommendationCard·JobToast**를 `components/ui`로 추출해 밀도 높은 화면을 일관되게.
- **밀도:** 프로 도구답게 조밀하게. 표 우선, 정보 위계는 타이포·색·간격으로.
- **에디터 톤:** 다크 기본(집중), 앱 나머지는 라이트 기본·다크 지원.

## 10. 스택 선택 & 근거 (큰 결정 기록)

> 큰 결정은 근거와 함께 기록하고 합리적 기본값으로 진행(멈추지 않음). 아래는 확정 기본값이며, 뒤집을 트리거를 함께 적는다.

### D1. 프론트엔드 스택 = STEPD와 동일 계열 ✅
**결정:** Next.js 16.2.3 · React 19.2.3 · TypeScript 5 · Tailwind v4 · shadcn/base-ui · lucide · `react-resizable-panels`(에디터) · `class-variance-authority`+`clsx`+`tailwind-merge`.
**근거:** ① 엔지니어가 v2 UI를 실 STEPD에 그대로 이식/역이식 가능 ② 디자인시스템·토큰 재사용 ③ 사용자 지시(동일 계열 권장). **대안 기각:** 새 프레임워크(Vite/Remix 등)는 이식성·연속성 손해.

### D2. 백엔드/DB = 재구축하지 않음, STEPD SPFN을 정본으로 재사용 ✅
**결정:** v2는 **UX(프론트) 재설계에 집중**. 백엔드(SPFN `@spfn/core` 잡 파이프라인·라우트)와 PostgreSQL 스키마는 **STEPD 것을 정본으로 재사용**하고, v2 프론트는 동일 도메인 모델을 타겟한다. 백엔드 실연결은 **후반 마일스톤(M6)**.
**근거:** ① 문제는 UX/구조지 백엔드가 아님 — 파이프라인·배포·인프라는 이미 성숙 ② SPFN은 **베타(`0.2.0-beta.45`) + WSL2 전용**(Windows 네이티브 미동작, `OPERATING_GUIDE §1.1`) + ffmpeg/Gemini/다중 Mac 노드 등 무거운 인프라 → 새로 세우는 건 위험·낭비 ③ 백엔드를 건드리지 않으면 운영 리스크 0(안전). **뒤집는 트리거:** 백엔드 API 계약이 v2 UX를 못 받쳐줄 때만 백엔드 확장 논의.

### D3. 초기 데이터 = 타입 안전 목(mock) 레이어 → 얇은 데이터 심(seam) ✅
**결정:** 스캐폴딩·UX 완성 단계는 STEPD 엔티티를 그대로 반영한 **인메모리 목 데이터 + 타입**(`lib/data`, `lib/types`) 위에서 구동. 데이터 접근을 **단일 인터페이스(repository/seam)**로 감싸 이후 실 API(RPC)로 교체.
**근거:** ① Windows에서 즉시 구동·클릭 가능한 프로토타입(WSL2/ffmpeg 불필요) ② UX 검증을 백엔드 준비와 분리 ③ seam 하나만 바꾸면 실연결. **대안 기각:** 처음부터 실 SPFN 연결 = 환경 세팅에 발목, UX 반복 속도 저하.

### D4. 배포/컷 엔진 = StepD 개념 흡수, 코드 미이식(초기) ✅
**결정:** 에디터 WYSIWYG·프리셋·자동자막·훅도구·이슈카드/리뷰흐름은 **UX로 재현**하되, 실제 ffmpeg/STT 렌더는 초기엔 목/스텁. STT(OpenAI)·세로 자막 번인·yt-dlp 임포트의 실동작은 후반에 STEPD 잡 확장 또는 사이드카로(통합분석 A/C안).
**근거:** 렌더 엔진 이식은 무겁고 백엔드 영역 → UX 먼저. **안전:** StepD 저장소는 읽기 전용, 절대 미수정.

### D5. 인증/RBAC ✅
**결정:** UI는 3역할(superadmin/admin/user) 게이팅을 처음부터 반영(목 세션). 실연결 시 `@spfn/auth`(STEPD와 동일)에 매핑.

## 11. 데이터 모델 매핑 (STEPD 엔티티 ↔ v2 타입)

v2 `lib/types`는 STEPD Drizzle 엔티티를 1:1 반영(정본 = STEPD). 주요:

| STEPD 엔티티 | v2 타입(개념) | 비고 |
|---|---|---|
| `programs` | `Program` | 메타·출연·장르·aiConfig·introOutro |
| `contents` | `Episode`(회차) | v2의 1급 객체(파이프라인 허브의 주어) |
| `sourceSets` / `sourceFiles` | `SourceSet` / `SourceFile` | master/clean variant, storageNode |
| `analysisResults` | `AnalysisResult` | 3-Pass 결과(JSONB) |
| `recommendations` | `Recommendation` | `kind`: candidate/short/clip, appeal, segments |
| `editProjects`(clips/encodes) | `EditProject` | `EditClip`/`Overlay`(v1→v2 좌표)/`Encode`(export/register 상태) |
| `clips` | `Clip` | clipType(T6 등)·status·thumbnailFileId |
| `media` | `Media` | 본편 영상 |
| `distributions` | `Distribution` | targetType+targetId+channel 유니크; **성과 컬럼 신규 여지** |
| `schedules` | `Schedule` | v2에서 recurrence/bulk/template 확장 |
| `smrFeeds`/`smrFeedMembers` | `SmrFeed`/`SmrFeedMember` | program/aggregate 모드 |
| `youtubeChannels`/`metaAccounts` | `Channel` 연동 | OAuth |
| `files`/`uploadSessions` | `FileAsset` | G-RAID/GCS |
| `processLogs` | `ProcessLog` | 상태 전이 이력 → v2 잡 센터 소스 |

- **신규(v2 표준):** 컨트롤드 보캐뷸러리 단일 모듈(`lib/constants`: clipType·clipCategory·targetAge·aspectRatio·pipelineStatus·distributionChannel). **계보(lineage)** 를 위해 `recommendationId → clipId → distributionId` 연결을 명시적으로 모델링(B3 해소).
