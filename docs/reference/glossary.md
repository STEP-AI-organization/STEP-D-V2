# 용어 정리 — 쇼츠 / 클립 / 하이라이트

> 2026-07-16. AENA(KT ENA 실서비스)와 STEP-D에서 공통으로 쓰는 콘텐츠 용어 정의.
> 외부 커뮤니케이션·시연·리포트에서 용어가 흔들리지 않도록 이 문서를 기준으로 한다.
> 코드 값(enum)의 단일 진실 소스는 `apps/web/src/lib/constants.ts`다.

## 계층 구조

```
원본 회차 (60~90분, 방송 마스터)
│
├── 쇼츠 (short)          = 숏폼
│     세로 9:16 · ~60초 · 훅 중심 재구성(순서 재배열 가능) · 썸네일 불필요
│     → YouTube Shorts, Meta Reels
│
└── 클립 (clip)           = 하이라이트   ← 사실상 동의어
      가로 16:9 · 수 분 · 서사 순서 유지한 압축 · 썸네일 필요 · 인트로/아웃트로 부착
      → 유튜브 일반 영상, SMR 계열(네이버TV 등)
```

## 핵심 정의

| 용어 | 정의 |
|------|------|
| **쇼츠** | 숏폼. "발견되는 콘텐츠" — 피드가 밀어주고 3초 안에 붙잡아야 함. 성과 지표는 완주율·임프레션당 시청시간 |
| **클립 (= 하이라이트)** | 롱폼 파생물. "선택되는 콘텐츠" — 썸네일·제목으로 클릭을 얻고 오래 붙잡아야 함. 성과 지표는 클릭률·시청 지속 시간. AENA 시스템 용어가 "클립", 방송사 실무 용어가 "하이라이트"일 뿐 같은 것 |
| **예고** | 클립의 하위 유형(다음 화 티저). 회차 **추천(kind) 레벨**에는 예고 구분이 없지만, 배포용 SMR `clipType`에는 이미 `T2`(예고편) 코드가 있다(아래 표). 엑셀 리포트의 "유형(쇼츠·하이라이트·예고)" 구분은 아직 수동 — 추천 단계의 예고추천은 향후 과제 |

## 선정 기준의 차이 (같은 재료, 다른 레시피)

- **분석은 회차당 1회** (STT·구간분석·점수화 — 발명신고서 구성 B 단일 원천 원칙).
- **쇼츠추천** = "순간"을 뽑는다 — 맥락 없이도 터지는 훅 (예: 리액션 5초). 길이 20~75초 제약.
- **클립(하이라이트)추천** = "이야기"를 뽑는다 — 시작~끝이 있는 서사 묶음 (예: 고백→노래→리액션 아크).
- **예고추천** (향후) = "떡밥 직전 컷" — 스포일러 직전에서 끊는 편집 문법. CX 서사 타임라인 필요.
- 최고의 쇼츠 구간이 하이라이트에는 조각으로만 들어가고, 좋은 하이라이트 아크를 쇼츠로 만들면 늘어진다 — 등호가 아닌 이유.

## 시스템 매핑

**추천 단위(kind)** — 프론트 정본은 `RECOMMENDATION_KINDS`(`apps/web/src/lib/constants.ts:109-114`) = `candidate`(후보) / `short`(쇼츠) / `clip`(클립).

| 개념 | AENA (aena repo) | STEP-D (이 repo) |
|------|------------------|------------------|
| 쇼츠 | `type: "short"` | `kind: "short"` |
| 클립=하이라이트 | `type: "clip"` | `kind: "clip"` |
| 비율 | `9:16-crop-main/sub`, `9:16-letterbox`, `16:9` | `ASPECT_RATIOS` 동일 값 (constants.ts:47-53) |
| 배포 채널 | — | `DISTRIBUTION_CHANNELS`: `smr`(네이버 SMR) / `youtube` / `meta` (constants.ts:101-106) |

> ⚠️ **내부 용어 드리프트**: 서버 `apps/server/src/pipeline.ts:16`의 로컬 `Recommendation` 인터페이스는 `"short" | "highlight"`를 쓴다 — 프론트 정본(`candidate/short/clip`)과 불일치. 실제 AI 추천 기록기(`apps/server/src/content-pipeline.ts:75`)는 `kind: "short"`만 기록하므로 당장 실무 영향은 작지만, 서버 코드를 만질 때 `highlight`를 정본 값으로 오인하지 말 것.

**배포용 SMR `clipType` (전 9종)** — 정의는 `constants.ts:10-20`. 추천→클립 채택 시 서버가 자동 부여: `apps/server/src/index.ts:506`(adopt 핸들러) `clipType: rec.kind === "short" ? "T6" : "TZ"`. (구 문서가 이 매핑을 "레거시 apps/api"로 표기했으나 오류 — apps/api에는 clipType이 없고, 현행 라이브 서버 코드다.)

| 코드 | 라벨 | 비고 |
|------|------|------|
| T2 | 예고편 | "예고"에 해당하는 배포 코드 — 이미 존재 |
| T3 | 촬영장 스케치 | |
| T6 | 숏폼 | = 쇼츠. adopt 시 `kind: "short"`가 여기로 |
| TZ | 클립영상 | adopt 시 쇼츠가 아닌 추천의 기본값 |
| TH | 하이라이트 | ⚠️ 하이라이트 코드는 TH다 (TZ 아님) |
| TI | 인터뷰 | |
| TS | 주요장면 기획 | |
| TT | 구작 | |
| T9 | 기타 | |

배포 필수 필드·채널별 준비도 판정은 [../plans/publish-fields-ux-plan.md](../plans/publish-fields-ux-plan.md) 참고.
